## 第二部分：软件与系统架构（The Software Stack）

### 2.0 软件架构概述

Leaf OS 是一套**从零自研的微内核操作系统**，运行于 Trdom 主处理器的最高异常级别（EL3）。它不基于 Linux，不 fork 自 Android，也不依赖任何开源内核——其调度器、IPC 机制、文件系统、驱动框架和 Hypervisor 全部由 Leaf Design OS Department 独立实现。

Leaf OS 与 Android 的关系被明确定义为 **“宿主 — 沙盒”** 模式：

- **Leaf OS（宿主）**：直接运行在硬件之上，拥有所有硬件寄存器的访问权限。负责 Trsec 通信、Trpower 电源策略、PACT 协议栈、OTA 安全更新等核心任务。
- **Android（沙盒）**：运行于 Leaf OS 内置的 **Type-1 Hypervisor** 之上。Android 看到的所有硬件（WiFi、蓝牙、GPS、传感器）均为 Leaf OS 提供的**虚拟设备节点**。Android 内核即使被 Root，也无法穿透 Hypervisor 访问 Trsec 或 Trcomm S1 的真实寄存器。

这一架构决定了 Leaf OS 的安全模型：**一切不可信，除非硬件直接证明其可信。**

### 2.1 启动链与引导程序

Tree 设备的启动链由下至上共分为五个层级，每一级在加载下一级之前均进行数字签名校验。任何一级校验失败，启动过程即终止，设备进入 TDFD 等待救援。

> PBL（固化在 Trdom ROM，不可刷写）
> ↓
> SBL（二级引导，加载 Trsec 并初始化 PMIC）
> ↓
> LBT（Leaf Boot Tool / 恢复模式，可选进入）
> ↓
> Leaf OS Kernel（微内核 + Hypervisor）
> ↓
> Android 容器（运行于 Hypervisor 之上）

#### 2.1.1 PBL（主引导只读存储器）

- **物理位置**：固化在 Trdom CPU 内部的 ROM 中，流片时写入，**不可刷写、不可修改、不可绕过**。
- **职责**：
  1. 上电后初始化 CPU 核心与时钟。
  2. 从 eMMC/UFS 的固定物理地址加载 SBL 镜像。
  3. 使用固化在 ROM 中的根公钥验证 SBL 的数字签名。
  4. 验签失败则直接进入 TDFD 模式（不加载任何外部代码）。
- **安全特性**：ROM 中的根公钥由 Leaf Design 根证书颁发，与 Trsec 的 PUF 根密钥无关联（双重信任根，防止单点失效）。

#### 2.1.2 SBL（二级引导程序）

- **物理位置**：存储在 eMMC/UFS 的 Boot1 分区，可通过 TDFD 更新，但**必须由 Trsec 授权**。
- **职责**：
  1. 初始化 Trsec 芯片，建立安全通信通道。
  2. 检查 Trsec 的 FUSE_BL 熔断位状态。
  3. 检测 BL 解锁状态。**若检测到已解锁，立即向 Trsec 发送熔断指令，烧断 FUSE_BL**。
  4. 加载 LBT 或 Leaf OS Kernel 镜像（视用户按键组合而定）。
  5. 验证所加载镜像的签名（使用 Trsec 提供的公钥）。
- **防回滚**：SBL 会读取 Trsec RPMB 中的 `MIN_TRUSTED_VERSION`，若待加载镜像版本号低于此值，拒绝加载并进入 TDFD。

#### 2.1.3 LBT（Leaf Boot Tool / 恢复模式）

LBT 是 Tree 设备的恢复模式，类比 Android Recovery，但功能与权限被严格收束：

**触发方式**：

- 关机状态下，同时按住音量上 + 电源键（组合键产线预置，不可更改）。

**允许的功能（白名单制）**：
| 功能 | 是否允许 | 说明 |
| :--- | :--- | :--- |
| 清除用户数据（双清） | ✅ 允许 | 常规恢复操作 |
| 清除缓存分区 | ✅ 允许 | 常规恢复操作 |
| 查看设备基础信息 | ✅ 允许 | 仅显示型号、SN、版本号 |
| LDB 查询类命令 | ✅ 允许 | 如 `ldb: query`、`ldb: reboot` |
| LDB 高危命令（shell/dd/mount） | ❌ 禁止 | USB 物理防火墙层拦截 |
| 卡刷 OTA 升级包 | ✅ 允许 | 必须使用 Leaf Design 签名的 OTA 包 |
| 刷入第三方镜像（类 TWRP） | ❌ 禁止 | 硬件写保护 + 启动自噬验证双重防护 |
| 进入 TDFD | ✅ 允许 | 需短接测试点 + 动态挑战（见 1.1.3） |

**安全机制**：

- **硬件写保护**：LBT 分区的 eMMC/UFS 写保护引脚（WP）由 Trsec 的 GPIO 直接控制。在 Leaf OS 正常运行时，WP 引脚拉高（写保护使能），任何软件层 `dd` 命令覆写 LBT 分区均被物理拒绝，返回 `WRITE_PROTECT_ERROR`。
- **启动自噬验证**：LBT 内核被加载到内存后，执行的第一条汇编指令是调用 Trsec 的 `AUTH_LBT` 硬件指令。Trsec 读取 LBT 所在物理分区的原始数据，实时计算 SHA-256，与镜像头部签名比对。比对失败则 Trsec 向 Trpower 发送硬线信号，**在 1 微秒内切断 CPU 核心供电**，无蓝屏、无日志、无劫持机会。
- **USB 物理防火墙**：LBT 模式下，USB 数据线（D+/D-）不直接连接 Trdom CPU，而是先经过 Trsec 的 USB MUX。Trsec 监听 USB 总线上的 SETUP 包，仅允许官方 LDB 白名单指令透传。检测到高危指令则物理层拉低 USB 数据线（Bus Reset），PC 端报“未知 USB 设备”。

#### 2.1.4 Leaf OS Kernel 加载

当 SBL 验签通过且未进入 LBT 时，系统加载 Leaf OS Kernel：

1. 初始化微内核调度器与内存管理单元（MMU）。
2. 初始化 Trsec 通信通道，验证 Trsec 固件完整性。
3. 根据 FUSE_BL 状态决定是否初始化 Trcomm S1 的 PACT 相关寄存器：
   - **FUSE_BL = 0（纯净状态）**：完整初始化 PACT 模块。
   - **FUSE_BL = 1（已熔断）**：PACT 内核模块直接返回 `-EPERM`，阻止 Trcomm S1 初始化，射频前端保持静默。
4. 启动 Type-1 Hypervisor，加载 Android 容器镜像。
5. 所有非关键外设（如触控、传感器、音频 Codec）在 Hypervisor 启动后才初始化，以缩短核心启动时间。

### 2.2 Leaf OS 微内核

#### 2.2.1 架构概述

Leaf OS 采用**微内核（Microkernel）架构**，内核态仅包含以下最小集合：

- 进程/线程调度器（优先级抢占式，支持硬实时）
- 内存管理（分页 + 段页式混合）
- 进程间通信（IPC，基于能力 Capability 模型）
- 硬件中断处理（支持不可屏蔽中断 NMI）
- Trsec 通信驱动（私有总线协议）
- Hypervisor 管理接口

**所有非核心服务**（文件系统、网络栈、USB 驱动、显示驱动等）均运行在**用户态**，通过 IPC 与内核通信。即使驱动崩溃，也不会导致整个系统重启。

#### 2.2.2 特权级定义

Leaf OS 定义了四个特权级别，层级从高到低排列：

| 级别    | 名称                           | 归属                 | 核心能力                            |
| :------ | :----------------------------- | :------------------- | :---------------------------------- |
| Ring -1 | 安全监控器（Secure Monitor）   | Trsec 固件           | 熔断控制、根密钥派生、硬件存证      |
| Ring 0  | 内核态（Kernel）               | Leaf OS 微内核       | 硬件调度、中断管理、Hypervisor 控制 |
| Ring 1  | 宿主服务（Host Services）      | Leaf OS 系统服务     | PACT 协议栈、OTA 更新器、电源策略   |
| Ring 2  | 沙盒管理（Sandbox Management） | Hypervisor           | Android 容器的生命周期与资源隔离    |
| Ring 3  | 用户态（User Space）           | Android 沙盒内的应用 | 受限系统调用，无法直接访问硬件      |

**关键设计**：Android 沙盒内的任何代码（包括 Android 内核）最高只运行在 Ring 2，**永远无法执行 Ring 1 及以上的指令**。即使用户获取了 Android Root 权限，也只是 Ring 2 内部的“土皇帝”，对 Leaf OS 宿主层无任何威胁。

#### 2.2.3 硬件调度与中断管理

Leaf OS 调度器支持**硬实时优先级继承**，确保 PACT 协议栈的音频流处理线程始终获得最高 CPU 时间片：

- PACT 音频接收线程：优先级 255（最高）
- PACT 信标扫描线程：优先级 254
- 普通 Leaf OS 系统服务：优先级 100-200
- Android 容器内所有线程：优先级 1-99（受 Hypervisor 限制）

**不可屏蔽中断（NMI）**：

- Trsec 发出的所有安全事件（如熔断触发、OTA 验证失败）均通过 NMI 通知 Leaf OS 内核。
- NMI 处理程序在 Trsec 专用 SRAM 中预置，即使内核因内存损坏而崩溃，NMI 处理逻辑仍可执行应急操作（如强制重启或冻结 IO）。

#### 2.2.4 IPC 与能力模型

Leaf OS 的 IPC 通信基于 **能力（Capability）模型**，不依赖传统的 UID/PID 权限检查：

- 每个进程在启动时获得一个能力列表（由父进程授予）。
- 调用 Trsec 接口需要持有特定的 `TrsecAccess` 能力。
- 能力无法被伪造或复制（由内核管理的唯一句柄）。

**Android 沙盒与宿主之间的 IPC**：

- Android 容器通过 Hypervisor 提供的 **虚拟 Binder 通道** 与 Leaf OS 系统服务通信。
- 所有从 Android 侧发往 Leaf OS 的请求（如“发起 PACT 求救”）均须经过 Leaf OS 的**策略引擎**审计：检查调用方是否为签名匹配的系统应用、是否持有有效的能力令牌等。
- 审计不通过则直接丢弃请求，不向 Trsec 或 Trcomm S1 转发。

### 2.3 Hypervisor 与 Android 沙盒

#### 2.3.1 Type-1 Hypervisor 架构

Leaf OS 内置的 Hypervisor 是**裸机型（Type-1）**，直接运行在 Leaf OS 微内核之上，负责管理和隔离 Android 容器。

**核心能力**：

- **硬件虚拟化**：所有硬件访问（内存、外设寄存器、中断）均被 Hypervisor 拦截并重定向到虚拟设备节点。
- **内存隔离**：Android 容器使用的物理内存页由 Hypervisor 分配，并在页表层面标记为“不可执行宿主代码”。
- **CPU 时间片限制**：Android 容器内的所有进程共享一个受控的时间片池，总 CPU 占用率不超过 40%（保证 Leaf OS 宿主服务和 PACT 协议栈的实时性）。

#### 2.3.2 虚拟设备节点

Android 看到的硬件设备均为 Leaf OS 提供的**虚拟设备节点（Virtio 类）**：

| 真实硬件         | Android 看到的虚拟设备     | 实际控制者                         |
| :--------------- | :------------------------- | :--------------------------------- |
| Trcomm S1 射频   | 虚拟 WiFi/BT/蜂窝适配器    | Leaf OS 网络代理                   |
| Trsec 安全岛     | 虚拟密钥库（Keymaster）    | Leaf OS 安全代理（仅转发受限请求） |
| Trneuro AI 芯片  | 不可见（Android 无法访问） | 禁用                               |
| Trpower 电源管理 | 虚拟电池电量计（只读）     | Leaf OS 电源策略引擎               |
| Carbon M 显示    | 虚拟帧缓冲（Framebuffer）  | Leaf OS 显示合成器                 |

**关键限制**：Android 无法直接调用 Trsec 的真实接口。所有涉及敏感操作（如加密签名、密钥派生）的请求必须通过 Leaf OS 的 `SecAuth` 服务代理，该服务会执行包名验证与动态令牌检查（详见第九部分）。

#### 2.3.3 Android 兼容层的 API 阉割

在 Android 沙盒的 `PackageManagerService` 和 Binder 驱动层面，Leaf OS 进行了定制化修改：

- 所有与 `TrsecManager`、`PactManager`、`PowerManager` 相关的 Binder 接口被标记为 `@SystemApi`。
- 非系统签名的应用调用这些接口时，Binder 驱动直接返回 `SecurityException`，请求**不会**传递到 Leaf OS 内核。
- 系统应用（如 Leaf Pay、Leaf Store）在编译时由 Leaf Design 官方签名，其 APK 中包含 Trsec 签发的**能力证书**，作为调用代理服务的唯一凭证。

#### 2.3.4 蜜罐诱骗参数

为增加 ROM 移植到非 Tree 设备的难度，Android 沙盒的 `ro.build.fingerprint` 和 `ro.product.model` 等系统属性中暴露的硬件参数（如基带型号、传感器厂商、GPU 型号）为**伪造值**。

这些伪造参数与 Leaf OS 内核读取的真实硬件 ID 不一致。极客若基于这些假参数移植 ROM 到普通骁龙设备，系统会在初始化阶段因硬件不匹配而 panic；若强行跳过，则 PACT 功能因 Trsec 缺失而永远灰掉。

### 2.4 文件系统与分区表

#### 2.4.1 分区布局总览

Tree 设备的 eMMC/UFS 存储空间按以下分区布局划分：

| 分区名称           | 大小     | 文件系统 | 挂载点          | 内容与用途                                     |
| :----------------- | :------- | :------- | :-------------- | :--------------------------------------------- |
| `bootloader`       | 4 MB     | 裸分区   | —               | SBL 引导程序（含防回滚版本号）                 |
| `lbt`              | 64 MB    | ext4     | /lbt            | LBT 恢复模式内核与 Ramdisk                     |
| `leaf_boot`        | 64 MB    | ext4     | /boot           | Leaf OS Kernel 与早期初始化脚本                |
| `leaf_system`      | 2 GB     | ext4     | /system         | Leaf OS 核心系统文件（微内核、驱动、系统服务） |
| `leaf_vendor`      | 1 GB     | ext4     | /vendor         | Trdom 私有驱动与硬件抽象层（HAL）              |
| `leaf_data`        | 4 GB     | ext4     | /data           | Leaf OS 应用数据与系统配置                     |
| `hypervisor`       | 512 MB   | ext4     | /hv             | Hypervisor 镜像与配置                          |
| `android_system`   | 4 GB     | ext4     | /android/system | Android 系统镜像（运行在容器内）               |
| `android_vendor`   | 2 GB     | ext4     | /android/vendor | Android 设备适配层（虚拟设备驱动）             |
| `android_userdata` | 剩余空间 | ext4     | /android/data   | Android 用户数据与安装应用                     |
| `pact_audit`       | 128 MB   | ext4     | （不可挂载）    | Trsec 诊断日志与审计记录（详见 2.4.2）         |
| `pact_tdfd_log`    | 64 MB    | ext4     | （不可挂载）    | TDFD 深度刷写日志与授权文件备份（详见 2.4.3）  |
| `pact_cache`       | 128 MB   | ext4     | （不可挂载）    | PACT 音频临时缓存（通话结束后清除）            |

#### 2.4.2 `pact_audit` 分区详解

- **物理位置**：固定在 eMMC/UFS 的末尾区域（最后 128 MB），由 Trsec 的 DMA 控制器直接管理。
- **访问权限**：
  - 写入：仅 Trsec 内部固件可写，Leaf OS 内核与 Android 沙盒均**无法挂载或读写**。
  - 读取：仅在进入 TDFD 且通过售后动态挑战后，Trsec 才允许将其映射为只读设备。
- **存储内容**：
  - Trsec Merkle 哈希链的完整副本。
  - 每次 PACT 呼叫的双轨音频存证（Track A + Track B 残差）。
  - OTA 更新记录（时间、版本、验签结果）。
  - 熔断事件记录（触发时间、触发原因）。
- **防擦除**：该分区的擦除命令被 Trsec 硬件层拦截，`blkdiscard` 或 `mkfs` 操作无法执行。

#### 2.4.3 `pact_tdfd_log` 分区详解

- **物理位置**：紧邻 `pact_audit` 分区后，大小 64 MB。
- **写入时机**：每次进入 TDFD 模式执行刷机操作时，Trsec 强制将以下信息加密写入该分区：
  - 使用的授权文件 Hash 与文件名。
  - 刷机时间戳（精确至微秒）。
  - PC 端 USB 主机 ID（若可读取）。
  - 授权文件内的微码动态挑战结果。
- **加密方式**：使用 AES-128-CTR，密钥由 Trsec UID + 温度传感器上电初值派生（弱于 PACT 的 SM4 加密，但足以阻止普通提取后的离线破解）。
- **OOBE 审计逻辑**：新机首次开机引导（OOBE）时，Leaf OS 强制读取该分区并解密。若存在未经云端备案的 TDFD 刷写记录，系统标记为“灰名单”，在 24 小时内若未连接企业内网进行“正版授权补签”，则触发熔断（烧断 FUSE_ABUSE）。

#### 2.4.4 防回滚与版本管理

Trsec 内部的 RPMB（Replay Protected Memory Block）区域维护了两个关键版本计数器：

| 计数器            | 用途              | 更新条件                                |
| :---------------- | :---------------- | :-------------------------------------- |
| `MIN_TRUSTED_VER` | 系统最小可信版本  | OTA 成功安装后递增，禁止刷回旧版本      |
| `MIN_PACT_VER`    | PACT 协议最小版本 | PACT 固件更新后递增，旧协议版本无法使用 |

**防降级攻击效果**：

- 极客通过 LBT 刷入旧版 Leaf OS 时，SBL 比较 `MIN_TRUSTED_VER`，旧版本号低于当前值则拒绝加载。
- 通过 TDFD 强制烧录旧版系统时，Trsec 在烧录完成后强制擦除所有用户数据分区，并在首次启动时重新生成全新的 PACT 根密钥（旧密钥作废），切断因版本回滚导致的授权文件绕过风险。

### 2.5 系统服务与守护进程

#### 2.5.1 核心服务列表

Leaf OS 内置以下核心系统服务，均运行在 Ring 1 级别（宿主服务域）：

| 服务名        | 职能                         | 依赖硬件               | 崩溃后行为                     |
| :------------ | :--------------------------- | :--------------------- | :----------------------------- |
| `SecAuthd`    | Trsec 通信代理与令牌验证     | Trsec                  | 系统重启                       |
| `Pactd`       | PACT 协议栈执行引擎          | Trcomm S1、Trsec       | 自动重启，不影响主系统         |
| `Powerd`      | 电源策略与 PACT-Cell 切换    | Trpower                | 系统重启                       |
| `OtaAgent`    | OTA 安全更新下载与预提交     | Trsec、Trcomm S1       | 自动重试，不影响系统           |
| `Auditd`      | 日志写入与 Merkle 哈希链维护 | Trsec、pact_audit 分区 | 立即触发 Trsec 告警            |
| `Hypervisord` | Android 容器生命周期管理     | Hypervisor             | Android 重启，Leaf OS 不受影响 |

#### 2.5.2 服务间通信

所有系统服务通过 Leaf OS 内核提供的**能力型 IPC** 进行通信：

- 每个服务启动时从父进程继承一组能力句柄。
- 调用其他服务的接口时，必须传递目标服务的接口能力。
- 非法调用（如未授权进程请求 `SecAuthd` 签名接口）在 IPC 路由层即被拦截，不会到达目标服务。

**第二部分结语**：
Leaf OS 的软件架构围绕 **“宿主 — 沙盒”** 隔离模型构建，所有安全敏感的硬件访问均被锁定在 Leaf OS 内核与 Trsec 协同控制的范围内。Android 容器在其中只占据一个受限的、可随时重置的虚拟空间——它的作用是兼容应用生态，而非承载系统安全。

从 PBL 的不可变 ROM 到 TDFD 的物理测试点，从 Hypervisor 的内存隔离到 `pact_audit` 分区的只追加日志，每一层设计都在回答同一个问题：**当攻击者拥有最高软件权限时，我们还能守住什么？** 我们的答案写在硅片里，写在熔丝里，写在物理不可克隆函数的随机分布里。

下一部分将进入 **Nature PACT 通信协议**，详细描述节点角色定义、连接建立流程、PACT-Racing 竞争仲裁，以及 PACT-Handover 平滑交接协议的完整时序。
