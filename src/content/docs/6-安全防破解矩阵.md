## 第六部分：安全防破解矩阵

### 6.0 模块概述

Leaf OS 与 Nature PACT 的安全模型基于一个核心假设：**攻击者可能拥有无限的时间和资源，可能物理接触设备，可能拥有 Root 权限，甚至可能逆向出 Leaf OS 的完整源码。** 在这一威胁模型下，任何纯软件防御都是脆弱的。因此，PACT 的安全矩阵被设计为**五层递进式物理防御**，每一层都依赖 Trsec 的硬件可信根，层层叠加，环环相扣。

五层防御的定位如下：

| 层级       | 防御对象                          | 核心机制                                        | 物理依赖                          |
| :--------- | :-------------------------------- | :---------------------------------------------- | :-------------------------------- |
| **第一层** | ROM 移植（适配非 Tree 设备）      | PUF 动态验证码 + 总线活性探测                   | Trsec PUF、Trcomm S1 硬件         |
| **第二层** | 二改 LBT（类 TWRP Recovery 攻击） | 硬件写保护 + 自噬验证 + USB 物理防火墙          | Trsec GPIO、Trpower 断电、USB MUX |
| **第三层** | 授权文件泄露与魔改                | CRL 吊销 + OTA 诱杀 + 私有 ISA 挑战 + OOBE 审计 | Trsec OTP、HSM、Trdom 私有指令集  |
| **第四层** | 物理拆解与芯片飞线                | 防拆走线 + TDFD 测试点动态变址                  | PCB 蛇形走线、Trsec 阻抗检测      |
| **第五层** | TDFD 深度刷写模式滥用             | 动态挑战-应答 + 防回滚 + 泄露文件即死           | Trsec + 云端 HSM 协同             |

五层防御之间无冗余依赖——任意一层被突破，后续层仍可独立生效。攻击者必须同时攻克全部五层，才能实现对 PACT 功能的完整绕过，这在物理层面几乎不可行。

### 6.1 防 ROM 移植（适配非 Tree 设备）

#### 6.1.1 威胁模型

极客从 Tree 设备中提取 Leaf OS 完整 ROM（含内核、驱动、系统服务），尝试刷入普通骁龙或联发科设备，以获得 PACT 功能。此攻击的动机通常是“白嫖”硬件能力或逆向分析协议实现。

#### 6.1.2 防御机制一：PUF 动态验证码

Trsec 的硬件身份基于物理不可克隆函数（PUF）。在 PACT 功能首次启用时，Leaf OS 向 Trsec 请求生成一段动态验证码：

```
验证码 = AES-256-Encrypt( RootKey, (当前RTC时间截断至小时 + 硬件随机数Nonce) )
```

- **RootKey**：由 PUF 响应值经模糊提取器（Fuzzy Extractor）生成，**不可读取、不可导出、仅 Trsec 内部可计算**。
- 验证码通过 Trcomm S1 的蜂窝或安全 WiFi 通道上传至 Leaf Design 云端备案服务器。
- 云端使用该设备 SN 对应的历史 RootKey 派生参数，独立计算期望验证码，与上传值比对。

**防移植逻辑**：非 Tree 设备没有 Trsec 芯片，无法生成基于 PUF 的验证码。即使通过软件模拟 RootKey，由于 PUF 响应是随机的且不可克隆，云端比对必然失败。

#### 6.1.3 防御机制二：总线活性硬探测

PACT 启动时，Trsec 向 Trcomm S1 和 Trneuro 的特定物理地址发送一个包含当前温度/电压值的**心跳加密包**，要求其在 **≤ 2 微秒** 内完成解密并返回响应。

- Trcomm S1 和 Trneuro 的硬件微码中内置了对此加密包的即时响应逻辑。
- 非 Tree 设备即使移植了驱动，但由于缺少对应硬件（或硬件寄存器地址不同），无法在 2μs 内响应。
- 超时或返回错误，Trsec 判定“环境不可信”，永久禁用 PACT 功能。

**移植者的困境**：即使通过 FPGA 模拟 Trcomm S1 的行为，2μs 的硬实时窗口也远低于通用操作系统（Linux/Android）的中断响应延迟（通常 > 50μs），无法通过软件手段模拟。

#### 6.1.4 防御机制三：Android 沙盒蜜罐参数

Leaf OS 向 Android 沙盒暴露的硬件属性（`ro.product.model`、`ro.hardware`、`ro.board.platform` 等）均为**伪造参数**，与实际 Trdom 硬件 ID 不一致。

- 极客若基于这些假参数移植 ROM，系统会在 Leaf OS 内核初始化阶段因硬件 ID 不匹配而 panic。
- 若攻击者绕过 panic，强行启动 Android 沙盒，PACT 相关服务在向 Trsec 请求令牌时仍因无硬件响应而失败。

### 6.2 防二改 LBT（类 TWRP Recovery 攻击）

#### 6.2.1 威胁模型

由于老旧版本的 Bootloader 不校验 LBT 分区，极客可通过内核提权漏洞（如 CVE 类型的本地权限提升）获取 Root 权限，然后用 `dd` 命令将二改 LBT 镜像覆盖至 LBT 分区，从而获得一个带完整 LDB 调试功能的恢复环境。进而通过 LDB 的高危命令（`shell`、`dd`、`mount`）绕过 Trsec 的部分验证逻辑。

#### 6.2.2 防御机制一：硬件写保护（WP 引脚锁定）

LBT 分区对应的 eMMC/UFS 物理写保护引脚（WP）由 Trsec 的 GPIO 直接控制：

- **常态（Leaf OS 正常运行时）**：WP 引脚拉高（写保护使能），任何软件层的写入命令均被 eMMC/UFS 控制器物理拒绝，返回 `WRITE_PROTECT_ERROR`。
- **唯一解锁时机**：设备通过物理按键组合进入**官方 LBT 模式**后，Trsec 在验证 LBT 自身签名合法后，临时释放 WP 引脚 3 秒供合法 OTA 写入。

**攻击后果**：即使极客获得 Root 权限并执行 `dd if=evil.img of=/dev/block/lbt`，该命令会在内核层面返回 I/O 错误，无法覆写任何字节。

#### 6.2.3 防御机制二：启动自噬验证

即使极客通过拆片烧录等方式强制覆写了 LBT 分区，在 LBT 内核被加载到内存的瞬间，仍会触发自噬验证：

1. LBT 内核的第一条汇编指令调用 Trsec 的 `AUTH_LBT` 硬件指令。
2. Trsec 读取 LBT 所在物理分区的原始数据，实时计算 SHA-256，与镜像头部预置的签名比对。
3. **比对失败**：Trsec **不返回任何错误码**（防止二改 LBT 捕获异常并伪造成功返回值），而是直接向 Trpower 发送一条硬线信号，在 **≤ 1 微秒** 内切断 CPU 核心供电。设备瞬间掉电，无蓝屏、无日志、无劫持机会。
4. **比对成功**：Trsec 释放 CPU 复位信号，允许 LBT 正常启动。

**攻击后果**：二改 LBT 即使被写入分区，也永远无法完成启动——它在第一条指令处就被物理断电。

#### 6.2.4 防御机制三：USB 物理防火墙

在 LBT 模式下，USB 数据线（D+/D-）不直接连接 Trdom CPU，而是**先经过 Trsec 的 USB MUX（多路复用器）**。

- Trsec 监听 USB 总线上的所有 SETUP 包。
- 仅允许官方 LDB 白名单指令（如 `ldb: query`、`ldb: reboot`）透传至 Trdom CPU。
- 检测到高危指令（如 `ldb: shell`、`dd`、`mount`），Trsec 立即在物理层将 USB 数据线拉低（Bus Reset），使 PC 端设备管理器报“未知 USB 设备”。

**攻击后果**：二改 LBT 即使侥幸存活（绕过前两道防御），其解锁的完整 LDB 命令也无法与 PC 通信——USB 信号在物理层被 Trsec 切断。

#### 6.2.5 防御机制四：秋后清算熔断

每次 LBT 自检失败（触发自噬断电）时，Trsec 会在 `pact_audit` 分区中记录一条“LBT 验签失败”日志，包含失败时间和尝试次数。

- 设备下次成功进入 Leaf OS 后，系统强制联网查询云端。
- 若该设备近期无官方售后维修记录，但存在 ≥ 3 次 LBT 自检失败记录，云端判定“存在未授权刷机尝试”。
- 云端下发定向熔断包，Trsec 烧断 FUSE_BL，永久锁定 PACT 功能。

**攻击后果**：即使攻击者在前三道防御中意外成功，最终的秋后清算仍会使其 PACT 功能永久失效。此机制让攻击者完全无法判断是哪一步操作触发了熔断，增加了攻击的心理成本。

### 6.3 防授权文件泄露与魔改

#### 6.3.1 威胁模型

Leaf Design 的售后与工程体系依赖两种授权文件：

| 文件类型                            | 权限级别 | 用途                          | 是否需账号                     |
| :---------------------------------- | :------- | :---------------------------- | :----------------------------- |
| **普通售后授权文件**                | 中       | LBT 卡刷 OTA、清除数据        | 需绑定工程师账号（动态令牌）   |
| **高权工程调试文件（Master File）** | 最高     | TDFD 深度刷写、Trsec 固件烧录 | **无需账号，仅凭文件即可刷机** |

风险在于：Master File 一旦泄露，任何持有该文件的人均可对任意 Tree 设备执行 TDFD 刷写，甚至绕过 Trsec 的部分验证。

#### 6.3.2 防御机制一：普通授权文件泄露 → CRL 实时吊销

- 每份普通授权文件在生成时均包含唯一 Serial 号和颁发时间戳。
- 云端维护一份 CRL（证书吊销列表）。一旦发现泄露，管理员立即将该 Serial 加入 CRL。
- Leaf OS 在 OTA 更新或 TDFD 握手阶段，会同步最新的 CRL 哈希。若文件 Serial 在 CRL 中，Trsec 直接返回拒绝码 `0xDEAD`。

**攻击后果**：泄露的文件在数小时内即告失效，攻击者无法利用其进行任何操作。

#### 6.3.3 防御机制二：Master File 泄露 → OTA 诱杀

Master File 泄露是最高风险事件，需以“诱杀”方式处置，而非简单拒绝（拒绝会让攻击者意识到文件已暴露，转而寻找其他漏洞）：

1. 云端通过静默 OTA 更新 Trsec 内部的“黑名单特征库”，将泄露文件的 SHA-256 和文件头部魔数写入硬件匹配引擎。
2. 当攻击者使用该文件在 TDFD 模式下尝试刷机时，Trsec **模拟加载成功**（让攻击者误以为绕过验证）。
3. 在刷写 Bootloader 分区的**最后 1% 时刻**，Trsec 向 Trpower 发送指令，将主 Flash 供电电压瞬时提升至 **6V（过压烧毁）**，或直接烧断 FUSE_BL 熔丝。
4. 设备永久变砖或 PACT 永久禁用，攻击者无法定位失败原因。

**攻击后果**：泄露的 Master File 不仅无法使用，还会毁灭使用它的设备。此机制极大震慑了内部泄密者。

#### 6.3.4 防御机制三：魔改防护 → 私有 ISA 微码动态挑战

极客可能尝试修改授权文件的内容（如替换 Hash 值或修改权限字段）。为此，授权文件内嵌一段**使用 Trdom 私有指令集（ISA）编写的可执行微码**：

1. TDFD 握手时，Trsec 向该微码发送一个 **32 位随机数种子**。
2. 微码必须在 **≤ 5 微秒** 内返回基于该种子的特定运算结果（如 CRC32 + 汉明重量）。
3. Trsec 比对计算结果。若正确，继续流程；若错误或超时，判定“文件被魔改”，触发熔断。

**防魔改内核**：Trdom 的私有 ISA 文档不公开，且每条指令的编码与 ARM/x86 完全不同。魔改者即使反编译出微码的二进制，也无法理解其语义，更无法修改后保持运算结果正确。

#### 6.3.5 防御机制四：OOBE 延迟审计

TDFD 刷写操作执行时，Trsec 强制将以下信息加密写入 `pact_tdfd_log` 分区：

- 使用的授权文件 Hash。
- 刷机时间戳（精确至微秒）。
- PC 端 USB 主机 ID（若可读取）。
- 刷机时的瞬时主板功耗曲线（作为硬件指纹辅助溯源）。

**OOBE 清算逻辑**：

- 设备首次开机引导（OOBE）时，Leaf OS 强制读取并解密该分区。
- 若存在未经云端备案的 TDFD 刷写记录，系统标记为“灰名单”，在 24 小时内若未连接企业内网进行“正版授权补签”，则触发熔断。

**攻击后果**：即使攻击者使用合法获取的授权文件进行了刷写，OOBE 审计仍可追溯其操作。若授权文件是盗用或泄露的，设备将在 24 小时内被熔断。

#### 6.3.6 防御机制五：指纹绑责

工程机在 TDFD 刷写前，强制要求操作人按压主板上的指纹传感器：

- 指纹的 Hash 值与授权文件绑定，一并写入 `pact_tdfd_log` 分区。
- 这意味着每一次非法刷机不仅留下了文件特征，还**物理锁定了具体操作人的生物特征**。

**威慑效果**：内部人员即使泄露了授权文件，也无法辩称“文件被盗”，因为指纹不可伪造。

### 6.4 防物理拆解与芯片飞线

#### 6.4.1 威胁模型

高阶攻击者可能拆开设备，用示波器或逻辑分析仪探测 Trsec 与 Trcomm S1 之间的总线信号，尝试截获密钥或模拟合法通信。更激进的攻击者可能用激光切割或酸蚀手段破坏封装，直接读取芯片内部存储。

#### 6.4.2 防御机制一：蛇形防拆走线

在主板上，包裹 Trsec 与 PMIC、Flash 之间关键信号线（EN 引脚、SPI 总线、I3C 数据线）的 PCB 走线设计为**蛇形防拆走线（Anti-Tamper Trace）**：

- 走线间距极窄（≤ 0.1mm），且分布于 PCB 内层（非表层），肉眼不可见。
- 一旦主板被钻孔、撕开分层、热风枪加热过度导致走线断裂，Trsec 检测到该线路上的阻抗突变。
- **触发后果**：Trsec 立即向 FUSE_TAMPER 熔断位写入 1，**零化所有 PACT 根密钥**，设备永久变砖且无法通过任何 TDFD 救回（因为 Trsec 已空）。

**攻击后果**：任何试图物理探测或飞线的操作都会触发自毁。攻击者必须在完全不触及防拆走线的前提下完成探测，这在物理上几乎不可能（因为关键信号线被走线包裹）。

#### 6.4.3 防御机制二：TDFD 测试点动态变址

TDFD 模式的触发需要短接主板特定测试点，但不同批次的 Tree 设备，其测试点的**逻辑电平序列（长按/短按/间隔时间）** 在产线烧录时随机生成并存入 Trsec OTP。

- 即使拆机找到了测试点的物理位置，不知道该批次设备的特定“敲击摩斯码”，也无法触发 TDFD 枚举。
- 该序列在 Trsec OTP 中加密存储，无法通过读 Flash 获取（OTP 区域受物理熔丝保护）。

**攻击后果**：物理拆机只能看到测试点的位置，无法获取触发序列。暴力穷举（尝试所有组合）在理论上可行但时间成本极高（若序列长度为 8 位，组合数为 2^8 × 时间间隔变量）。

### 6.5 TDFD 模式专属防护

#### 6.5.1 威胁模型

TDFD 是 Tree 设备最高级别的刷写通道，类比高通的 9008 模式。一旦被滥用，攻击者可烧录任意固件（含恶意修改的 Trsec 微码），绕过所有上层防御。

#### 6.5.2 防御机制一：动态挑战-应答

进入 TDFD 后，Trsec 强制进行动态挑战-应答握手：

1. Trsec 生成 32 字节随机挑战码，通过 USB 发送给 PC。
2. PC 端官方工具须将挑战码发送至云端 HSM，由 HSM 使用 Master 私钥签名后返回响应。
3. Trsec 使用预置的公钥验签。通过后授权刷写。

**防劫持**：挑战码包含当前 RTC 时间，验签时间窗口 ≤ 5 秒。抓包重放攻击在 5 秒后失效。

#### 6.5.3 防御机制二：防回滚

Trsec 内部 RPMB 维护 `MIN_TRUSTED_VER` 计数器：

- 每次合法 OTA 升级成功后，该计数器递增至当前版本号。
- TDFD 烧录固件时，Trsec 强制检查待烧录版本号 ≥ `MIN_TRUSTED_VER`。
- 若版本号更低，Trsec 拒绝烧录并记录“降级尝试”事件。

**攻击后果**：即使攻击者掌握了 TDFD 烧录能力，也无法通过刷回旧版本来绕过已修复的安全漏洞。

#### 6.5.4 防御机制三：泄露文件即死

若 `.tdfd` 授权文件被泄露并被加入黑名单，TDFD 烧录时：

1. Trsec 在加载 Loader 的前 0.5 秒匹配到黑名单，**不立即拒绝**（防止攻击者察觉）。
2. 在写入 Bootloader 分区的最后 1% 时刻，Trsec 向 Trdom CPU 发送“自杀指令”，烧断 CPU 内部 OTP 启动熔丝。
3. 烧录完成后，设备将永久停留在 TDFD 界面，无法引导任何系统（包括官方系统）。

**攻击后果**：泄露的 TDFD 文件不仅无法帮助攻击者获得设备控制权，反而会立即报废目标设备。此机制从根本上摧毁了授权文件的黑市价值。

### 6.6 五层防御的协同效果

五层防御在设计上**无依赖关系**——每一层独立生效，任意一层被突破不影响其他层的运行。这意味着攻击者必须**同时攻克全部五层**，才能实现对 PACT 功能的完整绕过：

| 攻击路径               | 需要攻克层数                                           | 物理可行性                       |
| :--------------------- | :----------------------------------------------------- | :------------------------------- |
| ROM 移植到普通骁龙设备 | 第一层（PUF 验证）                                     | 仅攻克一层即失败                 |
| 刷入二改 LBT           | 第二层全部四个子机制（写保护+自噬+USB防火墙+秋后清算） | 需同时绕过四个独立硬防           |
| 窃取并魔改授权文件     | 第三层全部五个子机制（CRL+诱杀+ISA挑战+OOBE+指纹）     | 需同时绕过五个独立硬防           |
| 物理拆解芯片           | 第四层 + 第五层部分                                    | 触发即自毁                       |
| 完整绕过 PACT 安全体系 | **全部五层**                                           | **实际不可行（物理熔断不可逆）** |

---

**第六部分结语**：

安全防破解矩阵的设计哲学可以用一句话概括：**“不信任任何软件，不依赖任何秘密，不给予任何第二次机会。”**

每一层防御都锚定在物理硬件上——PUF 的随机性、eFuse 的不可逆性、1μs 断电的硬实时性、蛇形走线的物理完整性。攻击者即使拥有 Leaf OS 的完整源码和最高 Root 权限，也无法绕过这些物理铁律。

PACT 的安全模型已经超越了“防御”本身——它的最终目标是：**让攻击的成本无限接近于重新设计一颗 Trsec 芯片，而那个成本，远超任何可能的收益。**

下一部分将进入 **Trsec OTA 更新与售后体系**，详细描述安全 OTA 管道、售后损坏定责机制、Trsec 替换防复制协议，以及 72 小时废除冷却期的完整流程。
