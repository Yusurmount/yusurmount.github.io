## 第五部分：电量管理与临终关怀

### 5.0 模块概述

在紧急场景中，电量是比信号更稀缺的资源。一台设备可能仍有满格信号，但若主电池耗尽，一切通信能力都将归零。PACT 协议的设计者从一开始就认定：**“求救”不应当因为“没电”而被终止。**

为此，Tree 设备在硬件层面引入了**双电池物理域隔离**架构，并在电源管理芯片（Trpower）中固化了**电量临终关怀（End-of-Life Care）** 策略。这套机制的核心目标可以概括为一句话：

> **当主电只剩 3%，当屏幕已经黑掉，设备依然有能力说：“我在这里。”**

---

### 5.1 双电池物理域隔离

#### 5.1.1 电池配置

Tree 设备内部包含两块物理独立的电池，它们在电气层面完全隔离，由 Trpower 芯片统一管理：

| 电池                       | 典型容量                                                   | 供电对象                                                                                    | 化学体系                          | 可独立工作               |
| :------------------------- | :--------------------------------------------------------- | :------------------------------------------------------------------------------------------ | :-------------------------------- | :----------------------- |
| **主电池（Main Battery）** | 常规容量（Pad 约 8000-10000mAh / Phone 约 4000-5000mAh）   | Trdom 主 CPU、Trneuro、Carbon M 显示芯片、屏幕背光、触控、传感器、音频 Codec                | 锂聚合物（Li-Po）                 | 否（依赖系统整体供电）   |
| **PACT-Cell**              | **500 - 1000 mAh**（视机型而定，Pad 取上限，Phone 取下限） | **Trcomm S1**（射频收发）、**Trsec**（加密/存证）、**Trpower 自身逻辑**、**麦克风偏置电路** | 锂锰电池（Li-MnO₂）或固态薄膜电池 | **是（完全独立供电域）** |

**PACT-Cell 的选型依据**：

- 锂锰电池（Li-MnO₂）或固态薄膜电池具有**极低自放电率**（< 1% / 年）和**宽温域工作能力**（-20°C 至 60°C）。
- PACT-Cell 在出厂时预充满电，日常使用中**不参与主电池的充放电循环**，仅在紧急时刻启用。其设计寿命与设备生命周期一致（≥ 5 年）。

#### 5.1.2 供电域物理隔离

Trpower 内部集成了两组独立的高侧 MOSFET 开关，分别控制主电池和 PACT-Cell 的输出通路。两组输出在物理层不共地，通过 Trpower 内部的**隔离式 DC-DC 转换器**分别稳压：

- **A 岛（主电池供电域）**：为 Trdom CPU、Trneuro、Carbon M、屏幕、触控、传感器等供电。
- **B 岛（PACT-Cell 供电域）**：专线为 Trcomm S1、Trsec、麦克风偏置电路供电。

**隔离意义**：

- A 岛短路或电压骤降不会影响 B 岛的正常工作。
- B 岛的电量不会被 A 岛上任何非关键外设（如屏幕背光）所消耗。
- 即使主电池被物理移除，PACT-Cell 仍能为 Trcomm S1 和 Trsec 供电。

### 5.2 3% 电量硬切换

#### 5.2.1 触发条件与检测机制

Leaf OS 的电源管理服务（`Powerd`）持续监测主电池电压与电量计读数。当满足以下**任一条件**时，触发硬切换：

1. **主电池电量 ≤ 3%**（由电量计 ADC 读数触发）。
2. **主电池电压低于 3.0V**（锂聚合物电池的放电截止电压，由硬件比较器触发，不受软件误差影响）。

**双重检测**：电压比较器是纯硬件逻辑，即使在 Leaf OS 内核崩溃、`Powerd` 服务未启动的情况下，Trpower 内部仍可独立触发硬切换。

#### 5.2.2 硬切换时序

硬切换过程由 Trpower 硬件状态机执行，全程不依赖软件，总耗时 < 100 微秒：

| 时间点  | 动作                                                                                | 执行者             |
| :------ | :---------------------------------------------------------------------------------- | :----------------- |
| T+0μs   | 电压比较器触发，向 Trsec 发送 EMG_PEND 中断                                         | Trpower 硬件比较器 |
| T+1μs   | Trsec 确认中断合法性（非误报），向 Trpower 回发 EMG_CONFIRM 指令                    | Trsec              |
| T+10μs  | Trpower 切断 A 岛（主电池）的 MOSFET 输出                                           | Trpower 硬件       |
| T+50μs  | Trpower 接通 B 岛（PACT-Cell）的 MOSFET 输出，专线供能 Trcomm S1 和 Trsec           | Trpower 硬件       |
| T+100μs | Trpower 向 Leaf OS 内核发送 NMI（不可屏蔽中断），通知系统“主电已切断，进入极限模式” | Trpower → Leaf OS  |

**关键特性**：在 T+10μs 至 T+50μs 的 40 微秒间隙中，Trsec 内部的小型超级电容（约 100μF）临时维持 Trcomm S1 的寄存器状态，确保射频不因瞬时断电而丢失握手信息。

#### 5.2.3 进入极限模式后的系统行为

一旦完成硬切换，Leaf OS 立即执行以下降级操作：

1. **屏幕与显示**：
   - 若主电池仍有残余电量（0% - 3%），屏幕转为单色模式，刷新率降至 1Hz，仅显示两行硬编码信息：
     - “紧急呼叫中 / SOS Active”
     - “剩余待机：XX 小时”（由 Trpower 估算）
   - 若主电池电压已不足以点亮屏幕，屏幕完全关闭，所有交互转至物理按键。

2. **交互切换**：
   - 触控层被切断，减少主电池消耗。
   - 音量上键被 Trsec 重映射为“重发求救信标”的硬线中断。
   - 音量下键被重映射为“挂断/终止通话”。
   - 电源键被重映射为“强制重启 Trcomm S1”（用于射频死锁恢复）。

3. **非关键服务冻结**：
   - Trneuro AI 修复芯片供电时钟被切断（若之前未切断）。
   - Android 容器被挂起（Hypervisor 暂停所有虚拟 CPU）。
   - 所有传感器（除加速度计用于运动唤醒外）关闭。

4. **射频策略调整**：
   - Trcomm S1 从“全双工语音模式”切换至“超低占空比信标模式”（若通话已结束）。
   - 发射功率降至最低有效值（仍可覆盖 10-30 米范围）。

### 5.3 极限生存模式（僵尸广播）

#### 5.3.1 定义与触发场景

当主电池完全耗尽（电压低于 2.8V，无法点亮屏幕或维持任何 A 岛负载）时，Tree 设备进入 **“极限生存模式（Extreme Survival Mode）”**，俗称“僵尸广播”。

此时设备的行为如下：

- **Trdom 主 CPU**：完全断电。
- **Android 容器**：完全断电。
- **Leaf OS 微内核**：完全断电。
- **Carbon M 显示芯片**：完全断电。
- **唯一保持工作的三颗芯片**：Trsec（安全岛）、Trcomm S1（通信协处理器）、Trpower（电源管理，仅 B 岛控制逻辑）。

#### 5.3.2 僵尸广播的信标策略

Trcomm S1 在极限生存模式下以**超低占空比**持续发送 PACT-Beacon：

| 参数         | 数值               | 说明                                                          |
| :----------- | :----------------- | :------------------------------------------------------------ |
| 广播间隔     | 5 秒               | 较常规 SOS 模式（3 秒）略长，进一步省电                       |
| 单次广播时长 | 200 ms             | 包含完整 OUI 特征 + GPS 坐标（若缓存有效） + 设备身份缩略哈希 |
| 发射功率     | 最低档（约 0 dBm） | 覆盖范围约 10-15 米（室内），3-5 米（穿墙后）                 |
| 占空比       | 4%                 | 200ms / 5000ms = 4%                                           |
| 平均功耗     | ≈ 0.8 mA           | 由 PACT-Cell 直接供电                                         |

**续航估算**：

- PACT-Cell 容量 500mAh（Pad 下限）÷ 0.8mA = **625 小时 ≈ 26 天**（理论值）。
- 考虑电池自放电、低温影响、Trsec 同时工作的耗电，实际续航约为 **6 - 8 天**（非 6-8 小时，此处纠正前期预估——经重新计算，500mAh 在 0.8mA 下可支撑 600+ 小时。前期“6-8 小时”为更保守场景下的极低电量阈值估算，现统一为 6-8 天）。
- 若 Trsec 同时进行音频缓存（麦克风持续拾音），续航缩短至 **约 48 小时**（2 天）。

#### 5.3.3 麦克风持续拾音与缓存

在极限生存模式下，Trsec 强制开启麦克风偏置电路，持续拾取环境声音：

- **音频编码**：Opus @ 4kbps（极低码率，仅保留基本可懂度）。
- **缓存策略**：音频数据加密后写入 Trsec 内部 SRAM（约 256KB）和外部 SPI NOR Flash（`pact_audit` 分区预留空间）。
- **缓存容量**：4kbps × 48 小时 ≈ 86 MB，由 SPI NOR Flash 的预留空间（约 128MB）容纳。
- **上传时机**：当附近有通信端路过并建立 PACT 连接时，Trsec 将缓存的音频摘要（最后 5 分钟）随信标一起发送。

**意义**：即使求救者已昏迷、无法主动说话，主电耗尽后的环境音（呼吸声、敲击声、搜救人员的呼喊）仍可被缓存并最终传递出去。

#### 5.3.4 退出极限生存模式

以下任一情况可退出极限生存模式：

1. **PACT-Cell 电量耗尽**：Trpower 检测到电压低于 2.5V，自动切断所有输出，设备彻底关机（不可逆）。
2. **外部充电接入**：插入 USB 充电器后，Trpower 优先为 PACT-Cell 充电（恢复至 3.0V 以上），然后尝试为主电池充电。充电到足以启动 Leaf OS 后，系统重新引导。
3. **被通信端唤醒**：通信端通过 PACT 信令发送“唤醒请求”，Trsec 评估后若判定为合法救援端，则短暂唤醒 Trdom CPU 的 RTC 域，点亮屏幕显示“救援已抵达”提示（若主电池仍有残余电量）。

### 5.4 反向馈能机制

#### 5.4.1 设计场景

在某些极端情况下，求救端的主电池和 PACT-Cell 均已耗尽，但通信端 Tree 设备仍有充足电量。此时，通信端可通过**无线反向充电**（基于 Qi 标准或 Tree 专属的近场谐振耦合）向求救端馈能，使其短暂恢复信标广播能力。

#### 5.4.2 触发条件

反向馈能仅在以下条件**全部满足**时触发：

1. 通信端已与求救端建立 PACT 连接（至少完成握手）。
2. 通信端主电池 ≥ 60%。
3. 求救端 Trpower 向通信端报告“双电耗尽，无法广播信标”。
4. 两设备物理距离 ≤ 5cm（近场耦合有效范围）。

#### 5.4.3 馈能过程

1. 通信端 Trpower 将无线充电线圈切换到“发射模式”。
2. 求救端 Trpower 将无线充电线圈切换到“接收模式”（即使主系统已关机，接收电路由 PACT-Cell 残余电压维持）。
3. 以 **500mW** 功率持续馈能 **30 秒**，向求救端 PACT-Cell 注入约 **15 J** 能量（≈ 4.2 mAh @ 3.7V）。
4. 此能量足以支撑求救端发送 **3-5 次完整的 PACT-Beacon**（含 GPS 坐标）。

**安全保护**：

- 馈能过程中，通信端 Trpower 实时监测自身电池温度，若温升 > 3°C 则立即终止。
- 求救端接收到的能量有限，无法为主电池充电（仅能激活 PACT-Cell 的欠压保护阈值），防止热失控风险。

### 5.5 电量管理的全链路场景示例

以下是一个完整的“主电耗尽 → 极限生存 → 救援抵达”示例流程：

| 阶段                    | 主电剩余 | PACT-Cell 剩余 | 设备状态                                 | 执行者              |
| :---------------------- | :------- | :------------- | :--------------------------------------- | :------------------ |
| 正常使用                | 100%     | 100%（未启用） | 全功能                                   | —                   |
| 触发 PACT 求救          | 15%      | 100%           | 通话中，AI 修复开启                      | Trcomm S1 / Trneuro |
| 主电降至 3%             | 3%       | 100%           | **硬切换触发**，屏幕单色，A 岛切断       | Trpower 硬件比较器  |
| 主电耗尽（0%）          | 0%       | 98%            | 屏幕黑掉，Trdom 断电，Android 挂起       | Trpower             |
| 极限生存（僵尸广播）    | 0%       | 98% → 逐步下降 | Trcomm S1 每 5s 广播一次，麦克风持续拾音 | Trsec + Trcomm S1   |
| 48 小时后（若未被发现） | 0%       | ≈ 20%          | 仍保持广播，但频率降至每 10s 一次        | Trsec 自适应调度    |
| 被附近通信端发现        | 0%       | ≈ 15%          | 建立 PACT 连接，传输缓存音频摘要         | Trcomm S1           |
| 救援抵达 / 外部充电     | 0%       | 进入充电状态   | 恢复主系统启动，显示救援提示             | Trpower + Leaf OS   |

**第五部分结语**：

电量管理的本质，不是“如何省电”，而是**“如何让最后那一点电，用在最该用的地方”**。

Trpower 的双域隔离设计，将“日常计算”与“紧急求生”从物理层彻底分开——主电池负责生活，PACT-Cell 负责生命。3% 硬切换不是应急策略，而是硬件级铁律；僵尸广播不是被动等待，而是主动宣告“我还在这里”。

当一台 Tree 设备的主电完全耗尽、屏幕完全黑掉时，它并没有死亡——它只是从“智能终端”降级为“求生信标”。而后者，在某些时刻，比前者重要得多。
