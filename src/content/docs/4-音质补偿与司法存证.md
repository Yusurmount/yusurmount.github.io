## 第四部分：AI 音质补偿与司法存证

### 4.0 模块概述

在 PACT 多跳中继场景中，音频信号每经过一次转发，都会引入压缩损耗、时延抖动和频段衰减。当跳数达到 2 跳以上时，通信端接收到的求救语音可能已破损至难以辨认——这直接威胁到"呼救"这一行为的根本目的：让对方听清你在说什么。

为此，Leaf Design 在 Tree 设备中集成了 Trneuro AI 神经修复芯片，并配套设计了**双轨制司法存证体系**。这两个模块共同构成 PACT 协议的"听觉皮层"与"法律基石"：

- **Trneuro AI 引擎**：实时修复破损音频，确保救援人员听得清。
- **双轨存证**：完整保留原始音频与修复音频，确保法庭上用得着。

两者缺一不可，互相独立，互不干扰。

### 4.1 Trneuro AI 修复引擎

#### 4.1.1 设计目标与约束

| 目标            | 指标                                                                   |
| :-------------- | :--------------------------------------------------------------------- |
| 修复延迟        | < 3ms（不干扰实时通话）                                                |
| 修复后 MOS 提升 | 至少 1.0 分（在原始 MOS ≤ 3.0 时）                                     |
| 功耗控制        | 动态调整，最低 0mW（旁路模式）                                         |
| 模型体积        | 基座模型 ≤ 2GB（适配 L4 SRAM Zone A）                                  |
| 训练数据        | 仅使用 Leaf Design 自采的多跳中继仿真语料（含 6 种语言、4 种噪声环境） |

#### 4.1.2 触发条件

Trneuro 仅在以下所有条件同时满足时激活：

1. **PACT 已触发**：通信端正在承担 PACT 通话。
2. **通信端主电池 > 40%**：避免 AI 修复争抢临终阶段的珍贵电量。
3. **当前跳数 ≥ 2**（或原始 MOS < 3.0）：一跳场景通常无需修复。

若通信端主电降至 20% 以下，Trpower 物理切断 Trneuro 的供电时钟（Clock Gating），系统强制回退至 PCM 透传模式——哪怕音质如"机器人对讲"，也必须优先保证链路不断。

#### 4.1.3 MOS 动态分级策略

Trneuro 内部固化了一个**无参考语音质量评估（Non-Intrusive MOS）** 硬逻辑。它不依赖参考信号，仅基于当前音频流的以下特征进行评分：

- 信噪比（SNR）估计。
- 频谱平坦度（衡量高频损失程度）。
- 包丢失率（PLR）与抖动缓冲区溢出频率。
- 语音活跃度（VAD）的连贯性。

**评分周期**：每 200ms 评估一次，动态调整 AI 干预策略。

| MOS 分数   | 干预策略                  | 硬件状态                                       | 功耗                          |
| :--------- | :------------------------ | :--------------------------------------------- | :---------------------------- |
| ≥ 4.0      | **完全旁路（Bypass）**    | Trneuro 数据通路直通，AI 核心休眠              | **0 mW**                      |
| 3.0 - 4.0  | **轻量级降噪 + 频谱均衡** | 仅激活 Zone B 中的 LFR（低频轮廓重建）模型     | **≈ 800 mW**                  |
| < 3.0      | **生成式插值补全**        | 激活 Zone A 基座模型 + Zone B 动态模型联合推理 | **≈ 3 W**                     |
| 间歇性断流 | **丢包隐藏（PLC）**       | 基于 RNN 预测丢失帧的 FFT 谱线                 | **≈ 1.5 W**（仅在丢包时触发） |

**策略切换的平滑性**：策略切换由硬件状态机完成，切换过程中不产生 pop 音或静音间隙。

#### 4.1.4 L4 级 SRAM 缓存分区（6-8GB）

Trneuro 独占的 L4 级 SRAM 被划分为三个物理分区，由硬件内存管理单元（MMU）直接管理：

| 分区                   | 容量 | 内容                                                         | 加载时机                                 |
| :--------------------- | :--- | :----------------------------------------------------------- | :--------------------------------------- |
| **Zone A（常驻区）**   | 2 GB | 基座模型（含 RNN 核心权重 + 频谱重构参数）                   | 系统启动时由 Trsec 验签后加载，永驻 SRAM |
| **Zone B（动态区）**   | 4 GB | 场景自适应模型（穿墙模型 / 高速移动补偿模型 / 风噪抑制模型） | 根据当前信道质量由 Trcomm S1 触发热加载  |
| **Zone C（乒乓缓冲）** | 2 GB | 音频数据包重排缓冲区 + FFT 中间结果缓存                      | 通话期间实时使用，每 200ms 轮换一次      |

**加载安全**：所有模型文件在加载前须经过 Trsec 签名校验。若验签失败，Trneuro 拒绝加载并回退至 Bypass 模式。

#### 4.1.5 与 Trdom 主 CPU 的协同分工

为降低 Trneuro 的推理负载，部分基础音频处理由 Trdom CPU 中的 DSP 核心承接：

| 处理任务        | 执行者                   | 说明                           |
| :-------------- | :----------------------- | :----------------------------- |
| 回声消除（AEC） | Trdom DSP                | Trneuro 不参与，减少流水线深度 |
| 噪声抑制（NS）  | Trdom DSP                | 仅做基础的平稳噪声抑制         |
| 频谱重构与插值  | Trneuro                  | 核心 AI 任务                   |
| 丢包隐藏（PLC） | Trneuro + Trdom DSP 协同 | DSP 做缓冲管理，Trneuro 做预测 |

通过这种分工，Trneuro 可将全部算力集中在"恢复可懂度"这一核心任务上，端到端修复延迟稳定在 3ms 以内。

#### 4.1.6 电量阈值联动

Trpower 向 Trneuro 提供三档供电状态，Trneuro 据此自动调整运行策略：

| 通信端主电剩余 | Trneuro 运行策略                                                        |
| :------------- | :---------------------------------------------------------------------- |
| > 40%          | 全功能开启，所有模型可加载                                              |
| 20% - 40%      | 强制降频（核心电压降低 15%），仅运行 Zone B 中的 LFR 模型，放弃高频修补 |
| < 20%          | Trpower 物理切断 Trneuro 供电时钟，系统强制旁路，AI 完全禁用            |

### 4.2 双轨制司法存证

#### 4.2.1 设计背景

AI 修复后的音频在法庭上可能面临"经过算法处理，不具备原始证据效力"的质疑。为解决这一法律困境，PACT 协议设计了**双轨并行存证**机制：

- **Track A（原始证据链）**：不经任何 AI 处理的 PCM 裸流，完整保留物理信道上的原始信号（含底噪、丢包导致的咔哒声、断续）。
- **Track B（辅助理解链）**：经 Trneuro 修复后的增强音频，或修复过程中生成的残差参数。

两者在 Trsec 内部被分别写入两个独立的环形缓冲区（FIFO），互不干扰，互不覆盖。

#### 4.2.2 硬件写入路径

音频数据从麦克风到存证分区的物理路径如下：

```
麦克风（PDM）
    ↓
Trcomm S1（采集 + 编码为 Opus 8kbps）
    ↓
Trsec（硬件复制门：数据流一分为二）
    ├── Track A（直写，不经 Trneuro）→ 原始 PCM 缓存
    └── Track B（转发至 Trneuro → 修复后返回 Trsec）→ 修复音频/残差参数缓存
```

**关键隔离**：Track A 数据从 Trcomm S1 到 Trsec 的 I2S 总线**不经过 Trneuro 的总线矩阵**，由 Trcomm S1 内部的硬件旁路开关（Bypass MUX）直连。即使 Trneuro 因过热死机或软件崩溃，Track A 仍能正常写入。

#### 4.2.3 存储格式与压缩策略

为节省有限的 PACT-Cell 电量和 `pact_audit` 分区空间，两条音轨采用不同的存储策略：

| 音轨            | 编码格式                  | 码率       | 存储内容                                                              |
| :-------------- | :------------------------ | :--------- | :-------------------------------------------------------------------- |
| Track A（原始） | Opus 8kbps（窄带）        | ≈ 1 KB/s   | 完整 PCM 裸流，含全部底噪与损伤                                       |
| Track B（修复） | 残差参数 + 基座模型版本号 | ≈ 0.3 KB/s | 不存储完整 WAV，仅存储 Trneuro 修复时生成的频段补偿增益、相位重构参数 |

**还原方法**：未来需要在法庭上播放 Track B 时，使用相同版本的 Trneuro 基座模型 + 残差参数进行离线重构。由于基座模型是公开审计的，残差参数是数学确定的，此过程可被第三方司法鉴定机构独立复现。

#### 4.2.4 存证生命周期

1. **通话中**：双轨音频实时写入 Trsec 内部的易失性环形缓冲区（容量约 10 分钟）。
2. **通话结束瞬间**：Trsec 将缓冲区内容冻结，生成 Merkle 根哈希，写入 `pact_audit` 分区的永久存储区。
3. **联网后**：`pact_audit` 分区中的哈希链通过加密通道上传至云端备案。
4. **司法调取时**：授权机构通过 TDFD + 售后动态挑战获得 `pact_audit` 分区只读权限，导出 Track A + Track B 残差。Track A 可直接播放，Track B 需通过官方工具重构。

#### 4.2.5 法理地位说明

在 Leaf Design 的《PACT 司法存证白皮书》中，双轨存证的法律定位如下：

| 音轨    | 法理定位                     | 法庭适用性                                                              |
| :------ | :--------------------------- | :---------------------------------------------------------------------- |
| Track A | **主证据（原始物证）**       | 可独立作为定责依据，无需任何 AI 辅助                                    |
| Track B | **辅助理解材料（听力辅助）** | 不可单独作为定责依据，须标注"AI 生成，置信度 XX%"，仅供救援过程还原参考 |

任何司法报告中提及 Track B 内容时，必须同时附上 Track A 的对应片段，并由鉴定机构签署《AI 修复未改变原始语义声明》。

### 4.3 Merkle 哈希链与防篡改日志

#### 4.3.1 哈希链结构

Trsec 内部维护的 Merkle 哈希链采用**追加式（Append-Only）** 写入模式。每一条日志条目（Log Entry）包含：

```
Entry_N = {
    timestamp:     uint64_t,    // 事件发生时间（微秒级）
    event_type:    uint8_t,     // 0x01=PACT呼叫, 0x02=OTA更新, 0x03=TDFD刷写, 0x04=熔断触发...
    event_data:    variable,    // 结构化载荷（如音频存证指针、固件版本号等）
    prev_hash:     uint8_t[32], // 前一条日志的 SHA-256 哈希值
    entry_hash:    uint8_t[32]  // 本条日志的 SHA-256（由 Trsec 硬件加速器计算）
}
```

**写入指针**：由 Trsec 内部的硬件 RTC 时钟驱动，每写入一条新日志，指针自动向后移动。Leaf OS 和 Android 均无法修改指针位置。

#### 4.3.2 存证完整性验证

当 `pact_audit` 分区中的日志被导出后，第三方鉴定机构可执行以下验证步骤：

1. 从第一条日志开始，逐条计算 `entry_hash = SHA256(timestamp + event_type + event_data + prev_hash)`。
2. 对比计算出的 `entry_hash` 与日志中存储的 `entry_hash` 是否一致。
3. 检查每一条日志的 `prev_hash` 是否等于前一条日志的 `entry_hash`。
4. 若链条完整且所有哈希匹配，则证明日志**自第一条起未被篡改**。

**防拆片篡改**：即使极客拆下 eMMC/UFS 芯片并用编程器修改了某条日志的 `event_data`，由于后续所有日志的 `prev_hash` 均基于旧值计算，修改后整个哈希链断裂。除非极客重算后续所有条目的哈希——但写入指针由 Trsec 硬件 RTC 驱动，无法回退，且 Trsec 内的黄金根哈希（第一条日志的锚点）固化在 OTP 中，无法覆盖。

#### 4.3.3 云端备案与时间戳公证

- 每次联网后，Trsec 将当前 Merkle 根哈希通过 Trcomm S1 的蜂窝通道（或安全 WiFi）上传至 Leaf Design 备案服务器。
- 备案服务器在收到根哈希后，加盖**国家授时中心同步的权威时间戳**并返回签名回执。
- 备案服务器不存储完整日志内容（保护隐私），仅存储根哈希与时间戳的绑定关系。

**司法意义**：备案时间戳证明了该哈希链在某个时间点之前已经存在且未被修改，构成"存在性证明（Proof of Existence）"。即使设备本地 `pact_audit` 分区被物理破坏，云端根哈希仍可作为证据完整性的辅证。

### 4.4 责任归属与免责证明

#### 4.4.1 求救端全责原则

PACT 协议的法律责任遵循**"谁求救，谁担责"**原则：

- 求救端设备（叶子节点）的 Trsec 对发出的求救信标进行数字签名。
- 签名中隐含了求救端唯一的 PUF 硬件身份。
- 若未来涉及"恶意求救"或"虚假报警"，仅需追查原始求救端的设备身份（由 Trsec 硬件锚定，无法伪造）。

**通信端免责依据**：

- 通信端仅作为"物理层耳麦代理"，不参与通话内容的产生或编辑。
- 通信端 Trsec 自动生成 `Relay-Proof`（纯转发存证），证明"本人未解密、未篡改、未存储任何音频"。
- 通信端若主动终止协助，在冷静期内有明确的取消与确认操作记录（同样写入 Merkle 哈希链），可作为"用户自主行为"的证据。

#### 4.4.2 中继端免责依据

分支节点（中继端）在转发过程中：

- 不解包音频载荷（仅操作 PHY 层信号）。
- 不存储任何音频数据。
- 每次转发均附加 Trsec 签名的 `Relay-Proof`，证明"仅做物理层放大，未接触内容"。

若未来涉及"通信链中某一环节被恶意利用"，中继端可通过 `Relay-Proof` 自证清白，且无需承担任何法律责任。

**第四部分结语**：

AI 修复与司法存证看似矛盾——一个在"改变"音频，一个在"固定"证据。PACT 协议通过**双轨并行 + 哈希链锚定**的设计，让两者从对立走向互补：Track A 守住法律的底线，Track B 打开救援的听阈。

Trneuro 修复的是声音的"可懂度"，Trsec 捍卫的是证据的"可信度"。两者共同构成了 PACT 协议在"听觉"与"法理"两个维度上的完整闭环。
