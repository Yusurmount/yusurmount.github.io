## 第七部分：Trsec OTA 更新与售后体系

### 7.0 模块概述

Trsec 作为 Tree 设备的安全根，其固件的更新管道和售后替换流程是整个信任链中最脆弱的两个环节：

- **OTA 更新**：若更新管道被劫持，恶意固件可植入后门，使所有硬件熔断机制形同虚设。
- **售后替换**：若替换流程存在漏洞，攻击者可利用售后渠道复制他人的 Trsec 身份，实现设备冒充或数据窃取。

为此，Leaf Design 为 Trsec 设计了**零信任 OTA 管道**和**黑盒派生售后体系**，确保固件更新的安全性不依赖任何外部因素，同时将售后工程师的操作权限降至“只动鼠标，不知原理”的盲操作级别。

### 7.1 安全 OTA 更新管道

#### 7.1.1 威胁模型

Trsec OTA 更新面临的主要威胁包括：

| 威胁         | 攻击方式                   | 潜在后果               |
| :----------- | :------------------------- | :--------------------- |
| DNS 劫持     | 将更新域名解析至恶意服务器 | 下载伪造固件包         |
| 中间人攻击   | 拦截并篡改固件包内容       | 注入后门代码           |
| 重放攻击     | 抓取合法固件包重复使用     | 降级至存在漏洞的旧版本 |
| 云端私钥泄露 | 攻击者获取签名私钥         | 签发任意恶意固件       |

#### 7.1.2 管道安全：硬编码 IP + TLS 1.3

Trsec 的 OTA 更新**不依赖 Leaf OS 或 Android 的网络栈**，而是通过 Trcomm S1 的基带直连蜂窝网络（或独立的安全 WiFi 通道）：

- **硬编码信标**：在 Trsec ROM 中预置全球多个 **Anycast IP 地址**（非域名），并固化根证书公钥。Trsec 联网时直接通过 IP 发起 TLS 1.3 连接，**彻底绕过 DNS 解析**。即使极客劫持了 Leaf OS 的 DNS，Trsec 的更新通道不受任何影响。
- **证书固化**：TLS 连接使用预置的根证书进行服务器身份验证，不依赖系统 CA 证书库。即使系统 CA 被篡改，Trsec 仍能识别合法服务器。

#### 7.1.3 防劫持：动态会话盐

每次 OTA 检查前，Trsec 的 TRNG（真随机数发生器）生成 32 字节的“会话盐（Session Salt）”，并通过基带通道（SMS 或小数据包）先行发送给云端：

1. Trsec 生成盐值 Salt，通过蜂窝通道发送给 Leaf Design OTA 服务器。
2. 服务器返回的固件包头部必须包含该盐值的 HMAC-SHA256 签名：`HMAC(ServerKey, Salt + 固件版本号)`。
3. Trsec 使用预置的服务器公钥验签。验签通过后才开始下载固件包体。

**防劫持效果**：劫持者即使截获了固件包，也无法实时篡改头部嵌入正确的盐值签名（因为盐值仅在 Trsec 内存中，未在网络传输，且每 5 分钟轮换一次）。

#### 7.1.4 防后门植入：A/B 分区 + 沙盒预提交

Trsec 内部 Flash 分为 **Active（运行区）** 和 **Standby（备用区）**：

1. **下载阶段**：新固件下载至 Standby 分区，**Active 分区在切换前不被修改任何字节**。
2. **预提交沙盒（10 秒）**：下载完成后，Trsec 将 CPU 核心切换至 Standby 区运行，但 **不将 Active 标记为失效**。在此沙盒中，Trsec 执行一组硬件自检指令：
   - 检查中断向量表是否指向合法地址（防恶意固件劫持中断）。
   - 模拟一次“熔断触发”信号，验证熔断控制逻辑是否异常（防后门抑制熔断）。
   - 检查 RTC 中断频率是否被篡改（防时间后门）。
   - 验证 Merkle 哈希链的写入指针是否正常（防日志篡改后门）。
3. **原子切换**：所有自检通过后，Trsec 在单个时钟周期内将 Active 指针切换至 Standby 分区。若任一自检失败，Standby 区立即被物理擦除，Active 区保持不变，Trsec 记录“固件验签失败”事件。

**防后门效果**：恶意固件即使在 Standby 分区被写入，在沙盒阶段即被自检逻辑捕获，永远无法进入 Active 运行区。

#### 7.1.5 防回滚：硬件版本号锁

Trsec 内部维护一个 **`HARDWARE_REV` 寄存器**（位于 OTP 区域），该寄存器只能递增，不可递减：

- 每次合法 OTA 升级成功后，`HARDWARE_REV` 递增至当前固件版本号。
- 任何低于 `HARDWARE_REV` 的固件版本（包括黑客伪造的旧版固件）在预提交阶段即被拒绝加载。
- 该寄存器由物理熔丝阵列实现，即使拆片也无法回退。

#### 7.1.6 核选项：蜂窝广播冻结令

极端情况——云端根私钥被攻破，黑客签发了大规模恶意固件：

1. Leaf Design 安全团队通过蜂窝广播（Cell Broadcast，即小区广播，非 SMS）向所有 Tree 设备发送一条“固件版本强制冻结指令”。
2. Trsec 收到后，立即将 `HARDWARE_REV` 强制提升至当前最新合法版本 + 1。
3. 任何低于此版本号的旧固件（包括黑客伪造的版本）均无法通过启动验证。
4. 旧版固件无法回刷——TDFD 模式下 Trsec 也会强制拒绝低于 `HARDWARE_REV` 的固件。

**效果**：蜂窝广播使用 3GPP 标准的小区广播信道，不受网络拥塞或个别基站劫持的影响。所有在网的 Tree 设备可在数分钟内被保护，恶意固件的生存空间被物理冻结。

### 7.2 售后损坏定责机制

#### 7.2.1 威胁模型

Trsec 可能因物理损坏（摔落、进水、雷击）或人为恶意破坏（激光攻击、探针探测、高压注入）而失效。售后需要准确区分“意外损坏”与“恶意破坏”，以决定保修责任归属。

#### 7.2.2 三级诊断日志

Trsec 内部维护三级诊断日志，分别对应不同的数据保真度与存储持久性要求：

| 日志级别              | 存储位置                                                      | 内容                                                                               | 持久性                                                 |
| :-------------------- | :------------------------------------------------------------ | :--------------------------------------------------------------------------------- | :----------------------------------------------------- |
| **Level 1（常规）**   | Trsec 内部 SRAM（易失）                                       | OTA 检查时间、固件包大小、签名状态                                                 | 掉电丢失                                               |
| **Level 2（异常）**   | Trsec 内部 OTP 区块（非易失）                                 | 电压波动记录、温度过冲事件、非法写入尝试（如 TDFD 测试点异常触发）                 | 永久保存，不可擦除                                     |
| **Level 3（毁灭性）** | Trsec 内部寄存器快照 + SPI NOR Flash（`pact_audit` 分区镜像） | Watchdog 复位前的最后 100ms 寄存器状态、总线时钟频率异常记录、防拆走线阻抗变化曲线 | 由 Trpower 独立电容维持 72 小时，之后写入 SPI NOR 固化 |

#### 7.2.3 损坏类型判定标准

售后工程师通过 **TDFD 模式（已授权）** 连接设备，向 Trsec 请求读取诊断日志。Trsec 返回加密数据，解密后包含以下判定依据：

| 损坏现象                     | 日志特征                                                                                        | 售后判定                           |
| :--------------------------- | :---------------------------------------------------------------------------------------------- | :--------------------------------- |
| **物理摔落/进水**            | 加速度计峰值记录 + 湿度传感器异常 + 供电电压**缓慢**跌落（< 0.1V/ms）                           | **非恶意损坏（保修）**             |
| **电压浪涌/雷击**            | PMIC 记录瞬间尖峰（> 6V，持续 10μs）+ 日志在尖峰时刻**突然中断**                                | **意外损坏（保修）**               |
| **恶意激光/探针攻击**        | 防拆走线阻抗突变记录 + 总线时钟频率异常（非标准 I3C 速率）+ 日志结尾出现“物理入侵警告码 0xE001” | **人为恶意损坏（拒保）**           |
| **试图刷入恶意固件导致变砖** | 日志显示“预提交自检失败（Failed at Checkpoint 3）”且循环重复 ≥ 3 次                             | **恶意操作损坏（拒保）**           |
| **未知原因且无日志记录**     | 日志尾部截断，无任何异常码                                                                      | **需送实验室深度分析（暂不判定）** |

#### 7.2.4 日志防篡改

每条日志条目都包含**前一条日志的 SHA-256 哈希值**，构成 Merkle 哈希链：

- 售后若试图伪造一份“意外损坏”日志替换“恶意损坏”日志，由于无法修改后续所有条目的哈希链，云端在验签时立即发现篡改。
- 日志读取过程中，Trsec 的硬件指针记录“已读偏移量”。售后完成后偏移量重置，但日志内容不可清零（防止攻击者通过多次读取拼凑）。

#### 7.2.5 用户申诉通道

若用户不认同售后判定，Leaf OS 在联网状态下为用户生成一份 **《Trsec 诊断报告摘要》**（不含私钥，仅含公钥签名的哈希值）：

- 用户可凭此报告向第三方司法鉴定机构申请复核。
- 鉴定机构使用官方公开的“日志解析工具（离线版）”解密日志，独立判断损坏性质。
- 此举确保售后判定过程透明，防止厂家一言堂。

### 7.3 Trsec 售后替换与防复制

#### 7.3.1 威胁模型

Trsec 售后替换面临的最大风险是：**拥有合法刷机权限的普通工程师，若勾结外部势力，能否利用售后流程复制出他人的 Trsec 身份？**

此威胁的关键在于：售后工程师拥有设备的物理接触权限和 TDFD 操作能力。若替换流程中暴露了 Trsec 的身份派生算法，则可能导致大规模身份伪造。

#### 7.3.2 核心原则：逻辑隔离 + 物理隔离

- **逻辑隔离**：高权工程师（Level-3 Core Engineer）**绝对不接触**通用 ROM，也**绝对不接触**空白芯片。他们只处理一串“已损坏芯片的序列号”文本。
- **物理隔离**：唯一 ROM 派生过程运行在**离线且物理锁定的 HSM（硬件安全模块）** 内，该设备独立于公司内网，仅有光纤单向输出（只能吐出派生文件，无法反向输入恶意代码）。

#### 7.3.3 黑盒派生流程

Trsec 的完整身份（含 PUF 根密钥、熔断状态位、加密证书）不是随机生成的，而是由一个 **“芯片主密钥（CMK）”** 通过 AES-256-CTR 派生而来：

1. **售后提交申请**：普通售后工程师通过售后 PC 工具，读取损坏设备的 TDFD 日志（证明芯片已无响应），自动提取 SN 和日志摘要哈希，生成加密申请包，发送至 L3 审批队列。
2. **高权审批**：L3 工程师在离线 HSM 上审批通过（只需点击“允许派生”）。
3. **HSM 派生**：HSM 内部维护一张表，存储着 **“序列号 → 当年生产时注入的派生盐值（Derivation Salt）”**。L3 审批后，HSM 调取该 SN 的历史盐值，使用通用固件骨架 + 该盐值重新计算所有加密密钥和证书，输出一个 `Trsec_Resurrection.bin` 文件。
4. **推送给售后工具**：该文件通过加密通道推送到售后 PC 工具。
5. **普通售后烧录**：工具自动加载该文件，通过 TDFD 模式烧入空白 Trsec 芯片。

**普通售后的知识边界**：全程只看到进度条，**看不到任何二进制、寄存器或密钥材料**。即使售后工程师试图拦截 `Trsec_Resurrection.bin` 文件，该文件也无法在其他设备或 PC 上使用——它绑定了目标设备的 SN 和烧录时间窗口。

#### 7.3.4 短后缀数学隔离（防双活）

新芯片的 SN 在原 SN 基础上追加 **`-R` 后缀（Replacement）**：

- 原 SN 为 `TREE-PAD-0x5A3F`，新芯片 SN 为 `TREE-PAD-0x5A3F-R1`。
- **密钥派生差异**：Trsec 的根密钥在派生时**强制混入后缀字符串**。因此，即使云端保留了旧 SN 的记录，旧 SN 的私钥与新 SN 的私钥**数学上完全不兼容**。
- **云端互斥锁**：云端数据库以“原始 SN（不带后缀）”为唯一主键。无论旧芯片或新芯片请求服务，云端查询主键：
  - 若旧芯片状态为 `Active`，新芯片状态为 `Pending_Activation`，云端**主动向新芯片发送“强制休眠”指令**，使其无法发起任何请求。
  - 反之，冷却期结束且未申诉后，旧芯片状态变为 `Revoked`，云端**向旧设备下发“落灰指令”**，云端拒绝与其握手。

**防双活效果**：在任何时刻，只有一个实体能获得云端的完整服务授权。数学隔离确保了即使旧芯片未物理损坏，也无法与新芯片同时存活。

#### 7.3.5 防内部勾结：多重物理边控

| 控制点         | 机制                                              | 作用                         |
| :------------- | :------------------------------------------------ | :--------------------------- |
| HSM 速率限制   | 同一 SN 在 **72 小时内只能派生一次**              | 防 L3 工程师批量生成复制芯片 |
| HSM 异常熔断   | 单日派生请求 ≥ 10 次，HSM 物理熔丝清零密钥材料    | 防内部大规模泄密             |
| 双因子烧录授权 | 烧录需云端令牌 + L3 工程师 OTP 双重确认           | 防单点沦陷                   |
| 操作日志透明   | 每次派生强制抄送 CEO 与安全总监邮箱（硬件级强制） | 防 L3 工程师私自审批         |

### 7.4 废除冷却期（72 小时三态迁移）

#### 7.4.1 设计背景

Trsec 替换流程中存在一个固有风险：**售后工程师可能误换芯片，或恶意攻击者通过社会工程骗取售后更换他人设备中的 Trsec。** 若替换操作是瞬发且不可逆的，用户将失去对自身设备证书主权的控制。

为此，Leaf Design 设计了 **72 小时废除冷却期**，将替换操作从“瞬发”变为“可逆、可申诉”的流程。

#### 7.4.2 三态定义

| 状态                         | 标识符 | 云端行为                                                         | 设备端表现                                                                 |
| :--------------------------- | :----- | :--------------------------------------------------------------- | :------------------------------------------------------------------------- |
| **Active（正常）**           | `0x00` | 允许所有云端服务（PACT 备案、OTA、证书签发）                     | 无异常提示，功能全开                                                       |
| **Pending_Revoke（冷却期）** | `0x01` | 拒绝新会话建立，但**保留旧证书有效性**；计时器开始 72 小时倒计时 | 每次联网后显示非阻塞气泡：“本机安全证书正在被替换，若非本人操作请点击申诉” |
| **Revoked（已废除）**        | `0x02` | 永久拒绝一切服务，证书进入全局 CRL                               | PACT 功能禁用，设备降级为普通平板/手机                                     |

**关键设计**：`Pending_Revoke` 状态下，旧设备仍可正常拨打电话、使用 4G/5G 网络。仅 PACT 相关的云端交互被挂起，但**本地 PACT 广播与中继功能完全不受影响**（因为本地 Trsec 未熔断）。

#### 7.4.3 新芯片的“待激活”状态

新芯片（带 `-R1` 后缀）的云端状态为 **`Pending_Activation`**：

- 在此状态下，新芯片**不能**用于任何 PACT 求救或签名操作。
- **激活条件**（满足任一）：
  1. 冷却期自然结束（72 小时后，无申诉）。
  2. 用户在旧设备上点击“确认更换，旧证书作废”。

若申诉成功，新芯片直接回退为 `Factory_Blank`（出厂空状态），需重新经历完整产线初始化流程才能再次使用。

#### 7.4.4 用户申诉流程

用户点击旧设备上的“申诉”后，触发**三重身份验证**：

1. **锁屏验证**：必须正确输入设备的锁屏密码/图案/PIN（售后工程师不知道）。
2. **SIM 卡绑定验证**：若设备插有 SIM 卡，系统向该号码发送 6 位验证码（通过蜂窝通道，不依赖网络）。
3. **生物特征验证**：若设备支持指纹/人脸，强制进行本地生物识别，并生成签名随申诉包上传。

**防骚扰**：同一 SN 在 30 天内最多发起 3 次申诉请求。超过 3 次后自动转入人工客服介入处理。

#### 7.4.5 离线场景兜底

若旧设备在 72 小时冷却期内全程未联网（如在无信号山区），云端倒计时以云端时间戳为准：

- 设备恢复联网时，若已超过 72 小时，系统**不自动将旧证书作废**，而是升级为**人工工单**：
  - 屏幕弹出提示“您的证书已过期，如需恢复，请联系官方客服（按 1 转人工）”。
  - 人工客服核实身份后，可手动解锁旧证书（不依赖自动流程）。
- 此机制防止用户因客观原因（如山区无信号）错过申诉期而遭受损失。

#### 7.4.6 冷却期交互文案

Leaf OS 提示文案经法务与 UX 双重打磨：

> **“安全提示：您的设备安全证书正在迁移至另一台设备。若您最近办理过售后维修，请忽略此提示。若非您本人操作，请立即点击此处进行申诉（剩余时间：XX小时XX分）。若超时未申诉，旧证书将自动作废以保护您的账户安全。”**

同时附上“查看更换申请详情”按钮，显示售后申请单号、申请时间、申请门店（由售后系统自动注入），使用户能快速对照是否为自己的真实维修记录。

### 7.5 售后体系全流程时序

以下为完整的 Trsec 售后替换示例流程（以用户本人正常维修为例）：

| 时间点       | 用户侧                                   | 售后侧                            | 云端状态                                                         |
| :----------- | :--------------------------------------- | :-------------------------------- | :--------------------------------------------------------------- |
| T+0h         | 送修损坏设备                             | 确认 Trsec 物理损坏，提交更换申请 | 旧芯片标记为 `Pending_Revoke`，新芯片标记为 `Pending_Activation` |
| T+0h ~ T+72h | 收到设备（带新芯片），开机看到冷却期提示 | 等待                              | 倒计时进行中                                                     |
| T+2h         | 用户确认“这是本人维修，确认更换”         | —                                 | 旧芯片标记为 `Revoked`，新芯片标记为 `Active`                    |
| T+2h + 1s    | 新芯片功能全开，提示消失                 | 流程完成                          | 云端记录“用户主动确认，冷却期提前结束”                           |

**若用户发现非本人操作（盗换）：**

| 时间点    | 用户侧                         | 售后侧                               | 云端状态                                        |
| :-------- | :----------------------------- | :----------------------------------- | :---------------------------------------------- |
| T+0h      | 被盗换设备被送修（用户不知情） | 恶意售后提交更换申请                 | 旧芯片 `Pending_Revoke`                         |
| T+1h      | 用户拿到设备，看到冷却期提示   | —                                    | 倒计时 71h                                      |
| T+2h      | 用户点击申诉，通过三重验证     | —                                    | 新芯片回退 `Factory_Blank`，旧芯片恢复 `Active` |
| T+2h + 1s | 提示消失，设备恢复正常         | 恶意售后操作被记录，安全团队介入调查 | 云端标记“申诉成功，存在可疑售后操作”            |

**第七部分结语**：

Trsec 的 OTA 更新与售后体系是 Leaf OS 信任链的最后两公里——前者负责“养”，后者负责“医”。两者的共同特征是：**不依赖任何人的善意，只依赖物理层的确定性。**

- OTA 管道不信任 DNS、不信任网络、不信任云端之外的任何实体。
- 售后替换不信任普通工程师的知识、不信任高权工程师的权限、不信任任何人的身份——只信任 HSM 的物理隔离和数学派生。

72 小时冷却期更进一步，将“芯片更换”这一系统行为转化为“用户可感知、可干预、可申诉”的透明流程，在安全与用户主权之间建立了最后的平衡。

下一部分将进入 **高级系统服务防护**，详细描述 Leaf Pay、Leaf Store 等高权限服务如何通过三因子动态令牌链防止包名伪装与权限滥用。
