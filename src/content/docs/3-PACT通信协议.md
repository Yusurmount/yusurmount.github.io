## 第三部分：Nature PACT 通信协议

### 3.0 协议概述

Nature™ PACT（Near-field Proximity Assistance & Coordinated Transmission，近域辅助与协同传输协议）是 Leaf Design TECHNOL Department 自主研发的**去中心化近域救援 Mesh 网络协议**。

PACT 不依赖任何中心服务器、基站、SIM 卡或互联网连接。它的核心运行逻辑是：**在无蜂窝信号或设备无基带的情况下，利用近场通信（WiFi Direct / 蓝牙）在 Tree 设备之间建立临时中继链路，由具备蜂窝能力的设备代为发起紧急呼叫。**

PACT 协议在设计上遵循以下四项基本原则：

| 原则           | 说明                                                                 |
| :------------- | :------------------------------------------------------------------- |
| **去中心化**   | 不依赖任何云端服务器或中心节点，所有决策在设备间本地完成。           |
| **硬件级可信** | 所有协议信令均经由 Trsec 签名与加密，软件层无法伪造或篡改。          |
| **最低功耗**   | 尽量将协议栈下沉至 Trcomm S1 硬件层执行，主 CPU 深度休眠。           |
| **平滑交接**   | 通话中任何一环（如通信端主动终止或信号恶化）均可无感切换至替代设备。 |

---

### 3.1 网络拓扑与节点角色

PACT 网络中的每一台 Tree 设备根据其**当前硬件状态**和**所处角色**，被动态划分为以下三种节点类型：

#### 3.1.1 叶子节点（Leaf Node / 求救端）

**定义**：无法独立发起紧急呼叫的设备。具体场景包括：

- 无基带的 Tree Pad（WiFi Only 版本）。
- 基带物理损坏的 Tree Phone。
- eSIM 配置失效或无法鉴权的设备。
- 有基带但当前无蜂窝信号覆盖的设备（此时它可先作为叶子节点求救，若附近有其他设备响应）。

**职能**：

- 发送 PACT-Beacon（含 OUI 特征码的求救信标）。
- 作为“耳麦”端：仅负责采集本地麦克风音频，并播放来自通信端的下行语音。
- 不参与蜂窝呼叫、不涉及 SIM 鉴权、不存储通话内容（通话内容由 Trsec 在通信端存证）。

**限制**：

- 叶子节点不参与中继转发（TTL 计数不增加）。
- 叶子节点的 Trcomm S1 在求救期间以最低占空比工作，节省 PACT-Cell 电量。

#### 3.1.2 分支节点（Branch Node / 中继端）

**定义**：无基带或无 SIM 卡、但具备完整 Trcomm S1 和 Trsec 的 Tree Pad。它不直接发起呼叫，但可承担**物理层信号再生**职能。

**职能**：

- 接收叶子节点的 PACT-Beacon，不解包、不解密。
- 在物理层（PHY）将原信号进行功率放大，并叠加 **跳数字段（Hop-Count）**（初始为 1，每经过一个中继端 +1）。
- 通过可重构增益天线以更高功率重新广播。
- **不接触语音内容**，不存储任何音频数据，仅转发信标与握手信令。

**限制**：

- 分支节点不参与 PACT-Racing 仲裁（不竞争成为通信端）。
- 分支节点不消耗 PACT-Cell 电量进行语音传输（仅信令转发，功耗极低）。
- 分支节点在转发时会附加 Trsec 生成的 **“纯转发存证（Relay-Proof）”**，证明“本人未解密、未篡改、未存储任何音频”。

#### 3.1.3 根节点（Root Node / 通信端）

**定义**：具备完整蜂窝基带（Trcomm S1 的蜂窝模块功能正常）、且有 SIM 卡或支持无卡紧急呼叫的 Tree Phone / Pad。

**职能**：

- 监听 PACT-Beacon 并响应。
- 通过 PACT-Racing 仲裁获得通信权后，与求救端建立辅助 WiFi P2P 音频链路。
- 利用自身基带发起 **3GPP 标准无卡紧急呼叫**（无需 SIM 鉴权）。
- 将求救端的上行音频混入本地 RTP 流，将基站下行的语音 RTP 包剥离 PCM 裸流推送给求救端。
- 负责通话全程的存证（Track A + Track B）生成与 Merkle 哈希链写入。

**限制**：

- 通信端在通话期间需保持 Trneuro AI 修复的可用性（电量充足时）。
- 通信端可在任何时候发起 Handover（主动退出），但须遵守冷静期协议。

### 3.2 连接建立流程

PACT 连接建立分为五个阶段，全程在 Trcomm S1 硬件层完成，主 CPU 无需唤醒。

#### 3.2.1 阶段一：求救端信标广播（Beaconing）

- **触发条件**：用户通过物理按键（长按音量+ 3 秒）或 Leaf OS 的 SOS 快捷入口触发 PACT 求救。
- **硬件动作**（由 Trcomm S1 执行）：
  1. 将 2.4GHz 射频切换至 PACT 专用信道（信道 6 或 11，由 Trsec 根据环境噪声动态选择）。
  2. 以 **5% 占空比** 发送 802.11 Beacon 帧，帧体包含：
     - **OUI 字段**：Leaf Design 注册的厂商特定 OUI（如 `00:1A:2B:3C:4D:5E`）。
     - **载荷类型**：`0x01`（PACT 求救请求）。
     - **求救端设备指纹哈希**（由 Trsec 生成，含 PUF 派生公钥的缩略）。
     - **当前 GPS 坐标**（若定位成功）。
     - **主电池剩余电量**。
  3. Beacon 帧**不含 SSID**，普通 WiFi 扫描列表不可见。

- **功耗**：每次 Beacon 广播约 200ms，间隔 3 秒，平均功耗约 1.5mA。

#### 3.2.2 阶段二：通信端被动嗅探（Sniffing）

- **硬件动作**（由通信端 Trcomm S1 执行）：
  1. 底层驱动持续在 2.4GHz 频段扫描特定 OUI。
  2. 命中后，**不唤醒主 CPU**，仅在 Trcomm S1 内部寄存器中缓存 Beacon 信息。
  3. 根据信号强度（RSSI）和跳数字段（Hop-Count）评估链路质量。

- **多端同时嗅探**：若多台 Tree 设备同时收到同一求救 Beacon，各自独立进入阶段三的仲裁流程。

#### 3.2.3 阶段三：PACT-Racing 竞争仲裁

当多台通信端同时响应同一求救时，通过 **硬退避（Hard Backoff）** 算法仲裁，决定由谁承担通信职责。

**仲裁因子计算**（由 Trsec 硬件完成）：

```
Priority_Factor = (100 - Battery_Percent) × 10 + (100 - |RSSI|)
```

- **Battery_Percent**：当前设备主电池剩余电量百分比（0-100）。
- **RSSI**：接收到求救 Beacon 的信号强度（dBm，取绝对值）。

**仲裁规则**：

- Priority_Factor 越小的设备，**响应优先级越高**。
- 计算完成后，Trsec 根据 Priority_Factor 生成一个 **微秒级延迟（Backoff Time）**：`Backoff(μs) = Priority_Factor × 50`
- 延迟最短的设备，其 Trcomm S1 会在第一个可用时隙发送 ACK 确认包。

**冲突解决**：

- 监听到其他设备先发 ACK 后，其余 Trcomm S1 在硬件层立即静默退让（Silent Mute），不再尝试连接。
- 整个过程耗时 **< 50 毫秒**，求救端主 CPU 完全无感知。

#### 3.2.4 阶段四：辅助 WiFi P2P 直连建立

- 获胜的通信端 Trcomm S1 向求救端发送 Probe Request（携带预置 PACT PSK）。
- 求救端响应后，双方建立 **WiFi P2P（Group Owner / Client）** 连接。
- 使用 5GHz 频段（若可用），通过 **802.11e（WMM）** 的 AC_VO 队列传输音频，确保语音优先级高于普通数据。
- 音频编解码锁定为 **Opus @ 8kbps**（窄带，兼顾低功耗与可懂度）。

**握手完成标志**：求救端 Trsec 生成会话密钥，并开始向通信端 Trsec 发送加密音频流。

#### 3.2.5 阶段五：蜂窝紧急呼叫发起

- 通信端 Trcomm S1 的基带部分强制发起 **3GPP 无卡紧急呼叫**：
  - 不读取 SIM 卡，不校验 IMSI。
  - 基站侧仅验证设备是否符合紧急呼叫频段要求。
  - 分配临时 TMSI，建立语音承载。
- 通话建立后，通信端将求救端上传的音频包混入上行 RTP 流，同时将下行 PCM 剥离后推送给求救端。

### 3.3 数据传输与限制

#### 3.3.1 音频传输协议

- **传输层**：RTP over UDP over WiFi P2P（不经过 IP 路由栈，直接通过硬件 MAC 地址通信）。
- **编码格式**：Opus @ 8kbps（窄带），每 20ms 打包一个帧。
- **抖动缓冲（Jitter Buffer）**：由 Trcomm S1 硬件维护，可抗 150ms 的网络抖动。

#### 3.3.2 链路保活与超时

- 双方 Trcomm S1 每 500ms 交换一次 **保活心跳（Keep-Alive Ping）**。
- 连续 5 次心跳无响应（即 2.5 秒），判定链路中断。
- 中断后求救端自动回退至 Beacon 广播状态，通信端释放蜂窝呼叫。

#### 3.3.3 TTL 生存时间限制

为防止求救包在 Mesh 网内无限循环导致广播风暴，每个 PACT 帧内嵌 **跳数字段（Hop-Count）**：

| 跳数 | 含义                                 |
| :--- | :----------------------------------- |
| 0    | 原始求救端发出的初始帧               |
| 1    | 经过一个分支节点（中继端）转发后的帧 |
| 2    | 经过两个分支节点转发后的帧           |
| ≥ 3  | **禁止继续转发**，Trcomm S1 丢弃该帧 |

**设计依据**：三跳足以覆盖绝大多数近域场景（相邻两堵墙以内的设备），超过三跳的音频质量已无实际可用性。

#### 3.3.4 端到端延迟保障

| 链路环节                  | 延迟预算               |
| :------------------------ | :--------------------- |
| 求救端麦克风采集 → 编码   | < 10ms                 |
| WiFi P2P 传输（一跳）     | < 20ms                 |
| Trsec 加密/解密           | < 5ms                  |
| Trneuro AI 修复（若启用） | < 3ms                  |
| 通信端蜂窝上行/下行       | < 40ms（基站侧不可控） |
| **总预算**                | **< 80ms**             |

### 3.4 PACT-Handover 平滑交接协议

#### 3.4.1 触发场景

Handover 过程可由以下任一情况触发：

1. **通信端主动终止**：用户点击 Toast 通知上的“终止协助”按钮。
2. **通信端电量告警**：通信端主电池降至 20% 以下，系统自动请求交接。
3. **通信端信号恶化**：通信端的蜂窝信号降至不可用阈值，自动发起交接。
4. **求救端主动切换**：求救端检测到当前通信端质量下降，主动发起重新广播。

#### 3.4.2 冷静期机制（10 秒窗口）

当通信端发起终止请求时，进入 **10 秒冷静窗口**：

- **T+0s**：通信端在 Toast 上显示“正在终止协助，10 秒后可取消”。
- **T+0s ~ T+10s**：
  - 主通话**不中断**，音频流持续传输。
  - 通信端 Trcomm S1 自动启动 **备用实例（Standby Instance）**：在主实例的 TDM 空闲时隙中广播“寻求接管者”的信标（携带当前通话的 Session ID）。
  - 求救端收到终止预兆后，也同步启动备用实例，准备与新的通信端握手。

**用户行为**：

- 若用户在 10 秒内点击“取消终止”，备用实例销毁，主通话恢复正常，日志记录“冷静期撤销”。
- 若用户点击“确认终止”或无操作，系统进入执行终止期。

#### 3.4.3 AI 交接摘要生成

在 T+10s 至 T+30s 的执行终止期间，Trneuro 芯片自动提取当前通话中的关键元数据：

| 元数据字段          | 说明                                                               |
| :------------------ | :----------------------------------------------------------------- |
| `cause`             | 求救原因（由 Trneuro 对音频内容进行语义分类：医疗/火灾/被困/其他） |
| `trend`             | 求救者状态趋势（如呼吸声强度变化、语音清晰度变化）                 |
| `last_location`     | 最后一次有效的 GPS 坐标                                            |
| `duration`          | 已通话时长                                                         |
| `participant_count` | 已参与的节点数量（含中继端）                                       |

摘要大小：约 **2KB**（结构化 JSON），经 Trsec 签名后附加在 Handover 信令中。

#### 3.4.4 硬截止与无感切换（30 秒）

- **T+30s**：主链路物理切断（无论是否找到备用设备）。
- **若备用实例已在 30 秒内握手成功**：
  - 新通信端接收 AI 摘要，无需重新询问求救原因。
  - 音频链路在新通信端与求救端之间重新建立，求救端感知为“短暂静音后恢复”，通话不中断。
- **若 30 秒内未找到备用设备**：
  - 求救端回退至初始 Beacon 广播状态，等待新的通信端响应。
  - 已生成的 AI 摘要暂存于 Trsec 本地，待新通信端响应后直接推送。

#### 3.4.5 交接责任链与存证

- 原通信端在切断链路前，生成 **《PACT 交接链证书》**，包含：
  - 原通信端 Trsec 签名。
  - 交接原因（用户终止/电量/信号/求救端请求）。
  - 新通信端的设备指纹（若已握手）。
- 该证书与双轨音频存证一并写入 `pact_audit` 分区，确保完整的可追溯性。

### 3.5 PACT 协议扩展：多跳中继（Relay Chain）

虽然协议规定 TTL ≤ 3，但中继链的建立方式遵循以下规则：

#### 3.5.1 中继端选择

- 分支节点（中继端）**不参与竞争**，只负责转发。
- 求救端接收到多个分支节点的转发信号时，选择 **RSSI 最强且 Hop-Count 最小** 的那条路径。

#### 3.5.2 中继端行为规范

- 分支节点收到求救帧后：
  1. 检查 Hop-Count < 3，若满足则进入转发流程。
  2. 在帧头部附加本机的 Relay-Proof（由 Trsec 生成）。
  3. 将 Hop-Count +1。
  4. 通过可重构增益天线以高功率模式重发。
- 分支节点不存储音频、不解密载荷、不参与后续通话。

#### 3.5.3 中继链断裂恢复

若中继端在转发过程中突然离线（如进入信号盲区）：

- 求救端在 2.5 秒内未收到通信端的下行音频，自动判定链路中断。
- 立即回退至 Beacon 广播状态，重新寻找通信端。
- 若仍有其他中继端存在，它们可能已缓存了求救帧并继续广播，无需求救端重新发送。

**第三部分结语**：

Nature PACT 协议的设计始终围绕一个核心场景展开：**当一台设备失去通信能力时，如何用最少的硬件资源、最短的路径、最可靠的加密，将“救命”这两个字传递出去。**

从叶子节点的低功耗 Beacon，到根节点的无卡紧急呼叫；从 PACT-Racing 的微秒级仲裁，到 Handover 的 30 秒无感交接——每一个协议字段、每一条超时规则、每一次硬件中断，都在为同一个目标服务：**让呼救这件事，在技术层面变得不可阻挡。**
