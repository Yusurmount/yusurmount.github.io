## 第八部分：高级系统服务防护

### 8.0 模块概述

Leaf OS 承载了 Leaf Pay（支付服务）、Leaf Store（应用商店）、Leaf Cloud（云同步）等高权限系统服务。这些服务运行在 Leaf OS 宿主层（Ring 1），拥有调用 Trsec 加密接口、读写 `pact_audit` 分区、管理 PACT 策略等敏感能力。

与此相对，Android 沙盒内运行着大量第三方应用。虽然 Hypervisor 已隔离了硬件访问，但存在一类攻击向量：**恶意应用通过声明与系统服务相同的包名（Package Name），尝试通过 Binder IPC 向 Leaf OS 宿主服务发起越权调用。**

在传统 Android 系统中，包名是权限检查的核心依据。但包名本质上是一个字符串——在 Root 权限下可被伪造。为此，Leaf OS 设计了**三因子动态令牌链（Tri-Factor Dynamic Token Chain）**，将包名信任从“字符串比对”升级为“硬件绑定的运行时活性证明”。

---

### 8.1 威胁模型

系统服务防护需要应对以下三类攻击：

| 攻击类型           | 攻击方式                                                                                                                          | 潜在后果                           |
| :----------------- | :-------------------------------------------------------------------------------------------------------------------------------- | :--------------------------------- |
| **包名伪装**       | 恶意应用在 AndroidManifest.xml 中声明与 Leaf Pay 相同的包名 `com.leaf.pay`，利用权限检查机制的字符串比对漏洞，调用 Trsec 加密接口 | 伪造支付授权、窃取 Trsec 签名密钥  |
| **进程注入**       | 通过 `ptrace` 或 `LD_PRELOAD` 注入合法系统服务进程，劫持其已获得的令牌，代发恶意调用                                              | 冒用合法服务的身份执行敏感操作     |
| **内核级 Rootkit** | 获取 Leaf OS 内核权限（Ring 0），伪造内核返回给 Trsec 的 PID 和包名查询结果                                                       | 绕过所有用户态验证，直接欺骗 Trsec |

三因子令牌链针对以上三类攻击分别设防，确保任一攻击路径在硬件层被阻断。

---

### 8.2 信任根：出厂白名单锚定

在 Tree 设备产线烧录阶段，Trsec 的 OTP 区域写入一张 **《系统服务白名单指纹库》**，包含所有高权限系统服务的身份信息：

| 字段                            | 说明                                | 示例                         |
| :------------------------------ | :---------------------------------- | :--------------------------- |
| 包名（Package Name）            | 系统服务的唯一包名标识              | `com.leaf.pay`               |
| 证书指纹（Certificate SHA-256） | Leaf OS 官方签名证书哈希            | `a1b2c3...`（64 位十六进制） |
| 基线 DEX 哈希（Base DEX Hash）  | 系统预置版本的核心 classes.dex 哈希 | `d4e5f6...`（64 位十六进制） |
| 最小版本号（Min Version）       | 允许调用 Trsec 接口的最低版本号     | `30001`（对应 v3.0.1）       |

**关键特性**：

- 此指纹库由 Trsec 私钥签名，烧录后不可修改。
- 由 Leaf OS 或 Android 沙盒均**无法读取或覆盖**此指纹库。
- 任何应用在请求 Trsec 敏感接口前，Trsec 硬件会强制读取该表进行第一轮“静态身份核验”。

### 8.3 三因子动态令牌链

令牌链由三个独立因子叠加构成，任一因子验证失败即拒绝调用：

| 因子       | 名称                             | 防攻击对象         | 验证者                   |
| :--------- | :------------------------------- | :----------------- | :----------------------- |
| **因子一** | 会话令牌（Session Token）        | 包名伪造、重放攻击 | Trsec 硬件               |
| **因子二** | 内核实名制（Kernel Real-Name）   | 用户态参数伪造     | Trsec + Leaf OS 内核协同 |
| **因子三** | 代码段审计（Code Section Audit） | 进程注入、钩子劫持 | Trsec 独立校验           |

#### 8.3.1 因子一：会话令牌（Session Token）

当 Leaf Pay/Store 等系统服务进程启动时，Leaf OS 内核（非 Android 层）向 Trsec 申请一个 **专属会话令牌**：

```
Token = HMAC-SHA256( RootKey, (PID + 真实包名 + 当前时间戳(截断至秒) + 硬件随机数) )
```

- **RootKey**：Trsec 的 PUF 根密钥，仅 Trsec 内部可计算。
- **PID**：进程 ID（由 Leaf OS 内核分配，用户态不可伪造）。
- **真实包名**：由 Leaf OS 内核从 `task_struct` 中直接读取，非应用传入参数。
- **时间戳**：截断至秒级，与云端时间同步（误差 < 5 秒）。

**有效期**：令牌 **60 秒** 后自动失效。服务进程在每次调用 Trsec 接口前，须重新申请新令牌。

**防重放**：每次令牌的随机数因子不同，抓包重放旧令牌在 60 秒后失效，且同一 PID 在同一秒内不可生成重复令牌。

**系统服务调用 Trsec 接口时的完整流程**：

1. 系统服务进程向 Trsec 发送请求，附带当前 Token。
2. Trsec **不信任传入的包名字符串**，而是执行以下操作：
   - 向 Leaf OS 内核发起 Hypercall，查询当前调用进程的真实 PID 和真实包名（内核从 `task_struct` 中直接读取）。
   - 用自身存储的 RootKey 重新计算预期 Token。
   - 比对预期 Token 与传入 Token 是否一致。
3. 若不一致，返回 `-EPERM`（操作不允许），并记录一次“令牌验证失败”事件。

#### 8.3.2 因子二：内核实名制（Kernel Real-Name）

因子一依赖内核返回的真实 PID/包名。若攻击者获取了 Leaf OS 内核权限（Ring 0），理论上可篡改内核返回的数据结构，向 Trsec 返回伪造的 PID 和包名。

为应对此威胁，Trsec 增加了 **硬实时超时陷阱**：

1. Trsec 在发出“查询 PID”的 Hypercall 后，启动一个**硬件看门狗定时器（5 微秒）**。
2. 若内核在 5μs 内未返回合法响应（或返回的数据格式异常），Trsec 判定“内核已被入侵”。
3. 触发后果：
   - 立即冻结所有涉及密钥导出的操作。
   - 将当前调用标记为“可疑”。
   - 向 Leaf OS 内核发送 NMI（不可屏蔽中断），强制重启 Leaf OS 内核（进入只读恢复环境）。

**防内核 Rootkit 效果**：

- 即使内核被 Rootkit 控制，5μs 的硬实时窗口使攻击者几乎不可能在中断上下文完成复杂的伪造操作。
- 通用 Rootkit 的中断处理延迟通常在 > 50μs 级别，硬实时窗口将其淘汰。

#### 8.3.3 因子三：代码段审计（Code Section Audit）

即使攻击者通过 `ptrace` 或 `LD_PRELOAD` 注入 Leaf Pay 进程，劫持其 Token 并代为发起调用，因子三仍能捕获此攻击：

1. Trsec 在接收调用请求时，要求 Leaf OS 内核提供调用进程的 **代码段校验和（Code Section Checksum）**：
   ```
   Checksum = SHA256( 进程的 .text 段起始地址 → 结束地址 )
   ```
2. Trsec 内部缓存了白名单中该系统服务的 **预期代码段哈希**（基于出厂基线 DEX 哈希计算）。
3. 若实际校验和与预期哈希不一致（说明进程被注入或篡改），即使 Token 正确，Trsec 仍拒绝服务。

**防注入效果**：

- `ptrace` 注入会修改进程的代码段映射，导致校验和变化。
- `LD_PRELOAD` 劫持会改变进程的内存布局，校验和同样变化。
- 攻击者无法同时维持“Token 正确 + 代码段哈希不变”两个条件。

### 8.4 权限分级与最小令牌原则

不是所有系统服务调用都需要完整的 Trsec 密钥访问权。Leaf OS 将调用类型分为四个级别，分别对应不同的令牌强度和验证流程：

| 权限级别           | 典型操作                          | 令牌要求                                  | 验证流程                            | 示例                        |
| :----------------- | :-------------------------------- | :---------------------------------------- | :---------------------------------- | :-------------------------- |
| **级别 0（低危）** | 读取设备序列号、查询 PACT 状态    | 仅需 Session Token（60s）                 | 因子一（基础校验）                  | 第三方应用查询设备信息      |
| **级别 1（中危）** | 发起 PACT 求救、修改 PACT 策略    | Session Token + 用户锁屏确认              | 因子一 + 因子二                     | Leaf OS 系统设置启用 PACT   |
| **级别 2（高危）** | 修改 Trsec 熔断策略、导出审计日志 | Session Token + 云端授权码 + 物理指纹按压 | 因子一 + 因子二 + 因子三 + 生物验证 | 售后维修、系统升级          |
| **级别 3（极危）** | 导出 Trsec 根密钥                 | **永久禁止**                              | 硬件物理拒绝                        | 任何尝试均被 Trsec 直接忽略 |

**级别 0-1 的调用**：由 Trsec 硬件独立判断，无需用户交互。
**级别 2 的调用**：必须同时满足以下条件：

1. Trsec 验证 Token 通过。
2. Leaf OS 弹出授权请求，用户确认（通过物理按键组合，非触控）。
3. 设备联网验证云端授权码（由 Leaf Design 服务器实时签发，5 分钟内有效）。

**级别 3 的调用**：Trsec 在硬件层面永久禁止导出根密钥。任何尝试读取根密钥的指令均被 Trsec 内部硬件比较器拦截，不返回任何数据（也不触发熔断——避免暴露“被探测”状态）。

### 8.5 证书吊销与密钥轮换

#### 8.5.1 证书泄露响应

若 Leaf Pay 的官方签名证书不幸泄露（恶意开发者用它重签名自己的应用）：

1. Leaf Design 安全团队通过 OTA 向所有 Tree 设备下发 **CRL 条目**（证书吊销列表增量更新）。
2. Trsec 在验证应用签名时，强制查询本地缓存的 CRL 黑名单。
3. 被吊销证书签名的应用（即使包名正确）在握手阶段即被 Trsec 拒绝，连获取 Session Token 的机会都没有。

**防降级攻击**：CRL 更新由 Trsec 独立的 OTA 通道完成，不受 Leaf OS 或 Android 版本影响。即使攻击者刷回旧版 Leaf OS，Trsec 内部的黑名单寄存器仍保持最新状态。

#### 8.5.2 双证书冗余

Leaf OS 出厂时预留 **两套根证书**（主证书 + 紧急备用证书）：

- 主证书正常使用。
- 备用证书仅存储于 Trsec OTP，不对外公开。
- 若主证书泄露，云端通过蜂窝广播指令切换至备用证书，Leaf OS 重新签名所有系统服务 APK 并推送 OTA。
- 切换过程无需更换 Trsec 芯片，仅更新 Trsec 内部的“有效证书标识位”。

### 8.6 开发者生态的透明桥接

为保障第三方应用开发者的正常使用，Leaf OS 在 Android 沙盒的 `PackageManagerService` 和 Binder 驱动层面进行了定制化修改：

- **API 阉割**：非系统签名的应用，在调用任何与 `TrsecManager` 相关的 Binder 接口时，Binder 驱动直接返回 `SecurityException`，请求**不会**传递到 Leaf OS 内核。
- **权限保护**：动态令牌申请接口 `getTrsecSessionToken()` 被标记为 `@SystemApi`，且受 `signature|privileged` 权限保护。普通应用在编译时即无法链接该 API（Android SDK 中已剔除该接口的 stub）。
- **开发者感知**：第三方开发者无法感知 Trsec 的存在——他们使用的标准 Android API（如 `KeyStore`、`Security`）的行为与普通 Android 设备一致。差异仅在底层由 Leaf OS 代理实现，但对开发者透明。

### 8.7 完整攻击防御示例

以下为一个完整的“恶意应用试图伪装 Leaf Pay”攻击的防御过程：

| 攻击步骤 | 恶意应用行为                                           | 防御机制                                                                                              | 结果                                                                                                           |
| :------- | :----------------------------------------------------- | :---------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------- | ------------ |
| 1        | 声明包名 `com.leaf.pay`，安装到 Android 沙盒           | Leaf OS 包名唯一性检查：系统分区已有同名 APK，安装被拒绝（由 Leaf OS 的 `PackageManager` 代理层拦截） | **无效**                                                                                                       |
| 2        | 修改 Android 内核，强行注册同名包名（绕过安装限制）    | 应用启动后，请求 `TrsecManager` 接口                                                                  | Binder 驱动检测到非系统签名，直接返回 `SecurityException`，请求未到达 Trsec                                    | **无效**     |
| 3        | 获取 Root 权限，向 Leaf Pay 进程注入 Shellcode         | Leaf Pay 进程的代码段哈希变化，调用 Trsec 时因子三校验失败                                            | Trsec 返回 `-EPERM`，记录“代码段篡改”事件                                                                      | **无效**     |
| 4        | 不注入代码，仅通过 ptrace 劫持 Leaf Pay 的 Token       | 劫持后，恶意进程以 Leaf Pay 的 PID 发起调用                                                           | Trsec 向内核查询真实 PID。内核返回 `task_struct` 中的 PID。恶意进程和 Leaf Pay 是不同的 PID，因此 Token 不匹配 | **无效**     |
| 5        | 获取内核 Rootkit，修改内核返回给 Trsec 的 PID 查询结果 | Trsec 发起 Hypercall 查询，启动 5μs 硬件看门狗                                                        | 内核 Rootkit 的篡改逻辑延迟 > 5μs，超时触发 NMI，内核强制重启                                                  | **攻击暴露** |
| 6        | （终极攻击）物理拆解 Trsec，试图读取根密钥             | 触发防拆走线断裂                                                                                      | FUSE_TAMPER 熔断，根密钥零化，设备永久变砖                                                                     | **设备报废** |

**结论**：恶意应用在任意攻击步骤均被阻断。不存在单一漏洞可连续突破三因子验证链。

**第八部分结语**：

高级系统服务的防护核心是：**不信任调用者告诉我的身份，只信任硬件和内核直接告诉我的身份。**

三因子动态令牌链将验证逻辑从“软件层字符串比对”下沉到“Trsec 硬件与内核的硬实时握手”，每一层验证都锚定在物理层——PUF 根密钥不可导出、5μs 硬件看门狗不可绕行、代码段哈希不可伪造。

而权限分级策略则进一步遵循“最小必要”原则：低危操作（如读取序列号）无需用户介入，中危操作（如 PACT 求救）需锁屏确认，高危操作（如熔断控制）需三重授权，极危操作（根密钥导出）则在物理层被永久禁止。

这套体系使 Leaf Pay 等系统服务的攻击面被压缩至最小，同时保持了正常用户的交互流畅性。
