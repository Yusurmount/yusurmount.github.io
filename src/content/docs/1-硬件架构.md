## 第一部分：硬件架构（The Hardware Stack）

### 1.0 硬件架构概述

Tree 系列设备（Tree Pad / Tree Phone）的硬件平台由六颗自研芯片共同构成，它们通过私有高速总线互联，在 Leaf OS 的调度下形成一套**无第三方依赖、全链路可控**的异构计算体系。

六颗芯片的职能划分遵循 **“单一职责 + 硬实时协同”** 原则：

| 芯片      | 职能                                 | 独立供电域         | 离线工作能力                   |
| :-------- | :----------------------------------- | :----------------- | :----------------------------- |
| Trdom 4/5 | 主应用处理（Leaf OS + Android 沙盒） | 主电池             | 否（依赖主电）                 |
| Trcomm S1 | 通信射频（WiFi/BT/蜂窝紧急呼叫）     | 主电池 + PACT-Cell | **是（PACT-Cell 可独立供电）** |
| Trsec     | 安全岛（身份/熔断/存证/令牌）        | 主电池 + PACT-Cell | **是（PACT-Cell 可独立供电）** |
| Trneuro   | AI 音质修复与推理加速                | 主电池             | 否                             |
| Trpower   | 电源管理与双域隔离                   | 主电池 + PACT-Cell | **是（PMIC 自身逻辑独立）**    |
| Carbon M  | 显示渲染与极限模式输出               | 主电池             | 否                             |

六颗芯片中，Trsec、Trcomm S1 和 Trpower 构成 **“求生铁三角”**——在主电耗尽后，三者仍可由 PACT-Cell 独立供电，维持信标广播与安全存证能力。

### 1.1 主应用处理器：Trdom 系列

#### 1.1.1 命名与迭代

Trdom 主应用处理器采用数字迭代命名，目前规划至 Trdom 5。各代之间保持架构兼容，后续数字递增代表制程提升、核心数增加或指令集扩展。

| 代际    | 制程 | 核心架构      | 定位       |
| :------ | :--- | :------------ | :--------- |
| Trdom 1 | 7nm  | 8核（1+3+4）  | 初代验证   |
| Trdom 2 | 6nm  | 8核（1+3+4）  | 能效优化   |
| Trdom 3 | 5nm  | 8核（2+4+2）  | 性能提升   |
| Trdom 4 | 4nm  | 10核（2+4+4） | 旗舰级     |
| Trdom 5 | 3nm  | 12核（3+5+4） | 下一代旗舰 |

#### 1.1.2 私有指令集架构（ISA）

Trdom 系列采用 Leaf Design 自研的私有指令集，**不完全兼容 ARM 或 x86**。这一设计的核心目的包括：

- **防止恶意代码跨平台移植**：针对 Trdom 的二进制程序无法在其他架构上运行。
- **授权文件微码加密**：高权工程文件中的动态挑战代码使用 Trdom 私有指令编写，魔改者若不解码私有 ISA 语义，无法通过 Trsec 的活性验证。
- **系统调用硬编码**：Leaf OS 与 Trsec 之间的 Hypercall 使用私有操作码，非 Tree 设备无法模拟。

#### 1.1.3 TDFD 深度刷写与下载模式

TDFD（Trdom Deeply Flash & Download）是 Trdom 系列独有硬件级刷写模式，层级高于 LBT（恢复模式），类比高通 9008 与 MTK 下载模式。

**触发方式**：

- 关机状态下，短接主板特定测试点并插入 USB。
- 测试点的逻辑电平序列（敲击摩斯码）在产线烧录时随机生成并存入 Trsec OTP，防止通用拆机触发。

**职能范围**：

- 烧录底层分区（含 PBL/SBL）。
- 救黑砖（LBT 分区损坏时）。
- 产线初始化与 Trsec 空白芯片身份注入。

**安全准入机制**：

- 进入 TDFD 后，Trsec 强制进行动态挑战-应答握手。
- PC 端须使用官方授权工具（含硬件加密狗）发送实时时间戳签名。
- 验签失败则 USB 数据总线被 Trsec 硬件隔离，读不到任何 Flash ID。

### 1.2 通信协处理器：Trcomm S1

#### 1.2.1 职能定位

Trcomm S1 是 Tree 设备所有射频通信的物理层执行者，覆盖 WiFi、蓝牙、蜂窝基带（含 3GPP 无卡紧急呼叫）三大通道。S 代表 **Signal**。

**核心设计原则**：

- **不依赖 Leaf OS 网络栈**：Trcomm S1 内置轻量级 MAC 层与基带协议栈，可在 Leaf OS 主 CPU 休眠时独立工作。
- **硬件级信标嗅探**：PACT 求救信标的识别在 Trcomm S1 的 ADC 采样流中以硬件逻辑门完成，零 CPU 唤醒。

#### 1.2.2 隐藏特征信标（OUI Beacon）

Trcomm S1 支持发送与接收**不含 SSID 的 802.11 Beacon 帧**，帧体中仅携带预定义的 OUI（组织唯一标识符）厂商特定字段。

**求救端（广播）**：

- 定时（每 3 秒）发送极短 Beacon 帧，帧体含 PACT 特征 OUI 与求救标志位。
- 不主动广播 SSID，常规 WiFi 扫描列表不可见。
- 发射功率与占空比由 Trpower 根据剩余电量动态调节。

**通信端（嗅探）**：

- Trcomm S1 底层驱动持续嗅探 2.4GHz 频段的特定 OUI。
- 命中后由 Trcomm S1 主动发起 Probe Request（携带预置救援 PSK），完成握手。
- 整个过程 Leaf OS 主 CPU 不参与，功耗低于蓝牙 LE 待机。

#### 1.2.3 可重构增益天线

Trcomm S1 内部集成可调谐孔径调谐器（Aperture Tuner），可动态调整天线阻抗匹配网络。

**两种工作模式**：
| 模式 | 阻抗匹配 | 场景 | 功耗 |
| :--- | :--- | :--- | :--- |
| 标准效率模式 | 50Ω 宽带匹配 | 日常通信 | 常规 |
| 高 Q 窄带谐振模式 | 定向高增益 | PACT 穿墙求救 | EIRP 提升 3-5dBm |

**频率选择**：PACT 模式下自动选择穿墙能力更强的 2.4GHz 频段（而非 5GHz），牺牲带宽换取覆盖范围。

#### 1.2.4 主/备双实例 TDM 时隙复用

在 PACT-Handover 冷静期，Trcomm S1 需同时维持主通话与备用实例搜索。通过 TDM（时分复用）硬逻辑实现：

- **主实例（优先级 1）**：占用 90% 射频时隙，用于实时音频 RTP 包传输。
- **备用实例（优先级 2）**：占用剩余 10% 空闲时隙，穿插发送 PACT-Beacon 嗅探帧。

两实例在物理层分时复用，互不干扰，求救端听不到任何杂音。

#### 1.2.5 无卡紧急呼叫

根据 3GPP TS 22.101 标准，Trcomm S1 的基带在检测到有效蜂窝信号时，**无需 SIM 鉴权**即可发起紧急呼叫（110/119/911）。

- 基站侧不校验 IMSI，仅分配临时 TMSI。
- 呼叫全程由通信端 Trcomm S1 完成，求救端仅传输音频流。
- 通话内容不经过 SIM 卡或 eSIM 模块，彻底绕开“SIM 无效/欠费/损坏”障碍。

### 1.3 安全岛：Trsec

#### 1.3.1 职能定位

Trsec 是 Tree 设备的安全根与信任锚，独立于 Trdom 主 CPU 运行，拥有自己的 CPU 核心、SRAM、OTP 和加密协处理器。它负责：

- 硬件身份生成与验证（PUF + eFuse）
- 熔断机制执行与状态管理
- 加密存证与 Merkle 哈希链维护
- 动态令牌生成与系统服务鉴权
- OTA 固件安全校验

**关键承诺**：Trsec 的固件和密钥材料**不可被 Leaf OS 或 Android 沙盒读取**，所有敏感操作均在 Trsec 内部完成，仅返回结果。

#### 1.3.2 PUF（物理不可克隆函数）

Trsec 的硬件身份基于芯片制造过程中产生的**硅片随机掺杂差异**（物理不可克隆函数）生成。

- **根密钥派生**：PUF 响应值经模糊提取（Fuzzy Extractor）处理后产生 256 位根密钥（Root Key）。
- **动态验证码**：PACT 启动时，Trsec 使用根密钥 + 当前 RTC 时间 + 硬件随机数生成一次性验证码，云端使用私有算法验签。
- **不可复制性**：即使同一片晶圆上相邻的两颗 Trsec 芯片，其 PUF 响应也不同。物理克隆在数学上不可行。

#### 1.3.3 eFuse 物理熔丝与熔断机制

Trsec 内部包含多组一次性可编程（OTP）电熔丝阵列（eFuse），用于记录不可逆的状态位。

**主要熔断位**：
| 熔断位 | 触发条件 | 后果 |
| :--- | :--- | :--- |
| FUSE_BL | Bootloader 解锁（解锁 BL） | PACT 功能永久禁用 |
| FUSE_TAMPER | 防拆走线断裂或物理入侵探测 | 零化根密钥，设备永久变砖 |
| FUSE_ABUSE | PACT 验证连续失败 3 次（1 小时内） | 锁定 PACT-Cell 放电电路 |
| FUSE_MASTER | 加载泄露的高权 Master 文件（OTA 诱杀） | 烧断 CPU OTP 启动熔丝，卡 TDFD 界面 |

**特性**：所有 eFuse 位**只能从 0 烧为 1，不可逆**。即使 Leaf OS 内核被 Root，软件层也无法回写或清除熔断状态。

#### 1.3.4 Merkle 哈希链与防篡改日志

Trsec 内部采用 **Merkle 树哈希链** 结构记录所有安全事件（PACT 呼叫、OTA 更新、TDFD 刷写、验证失败等）。

- **写入规则**：只追加（Append-Only），每条新日志包含前一条日志的哈希值。
- **物理指针**：写入指针由 Trsec 硬件 RTC 驱动，软件层无法移动。
- **校验码**：通话结束或事件完成时，生成 Merkle 根哈希并签名存储。
- **联网上报**：重新联网后，将哈希链上传云端备案。任何篡改（即使物理拆片改 Flash）都会导致根哈希验签失败。

#### 1.3.5 动态令牌与系统服务鉴权

Trsec 为 Leaf Pay、Leaf Store 等高权限系统服务提供硬件级令牌验证，防止第三方应用通过包名伪装调用 Trsec 接口。

令牌生成公式：
**Token = HMAC-SHA256( RootKey, (PID + 包名 + 当前时间戳(秒级) + 随机数) )**

- 令牌有效期 60 秒，绑定调用进程 PID。
- 验证时 Trsec 主动向 Leaf OS 内核查询调用方真实 PID 与包名（用户态不可伪造）。
- 超时（>5μs）或返回异常则判定内核遭入侵，触发安全降级。

### 1.4 AI 神经修复芯片：Trneuro

#### 1.4.1 职能定位

Trneuro 是专为 PACT 多跳中继场景设计的 **AI 音质修复加速器**，负责在通信端重建因跳数增加或信号劣化而受损的音频内容。

#### 1.4.2 硬件规格

| 项目     | 参数                                                                   |
| :------- | :--------------------------------------------------------------------- |
| 架构     | 轻量级 RNN 硬核（无参数量化）                                          |
| 独占缓存 | L4 级 SRAM，容量 6-8GB                                                 |
| 分区策略 | Zone A（基座模型 2GB）+ Zone B（动态模型 4GB）+ Zone C（乒乓缓冲 2GB） |
| 延迟     | 修复流水线 < 3ms                                                       |
| 功耗     | 旁路 0mW / 轻量 800mW / 生成式 3W                                      |
| 触发条件 | 通信端主电 > 40%，且 PACT 已触发                                       |

#### 1.4.3 MOS 动态分级策略

Trneuro 内部固化 **无参考语音质量评估（Non-Intrusive MOS）** 硬逻辑。每 200ms 对入耳音频进行一次评分：

| MOS 分数  | AI 干预策略           | 描述                                |
| :-------- | :-------------------- | :---------------------------------- |
| ≥ 4.0     | 完全旁路（Bypass）    | 音频质量良好，AI 休眠，透传数据     |
| 3.0 - 4.0 | 轻量级降噪 + 频谱均衡 | 修复细微瑕疵，功耗约 800mW          |
| < 3.0     | 生成式插值补全        | 波形重构，恢复可听度，功耗拉满至 3W |

**电量阈值联动**：

- 主电 > 40%：全功能开启。
- 20% - 40%：强制降频，仅运行低频轮廓重建（LFR）模型。
- 主电 < 20%：PMIC 切断 Trneuro 供电时钟（Clock Gating），回退至 PCM 透传。

### 1.5 电源管理芯片：Trpower

#### 1.5.1 职能定位

Trpower 是 Tree 设备的能量调度官，负责主电池与 PACT-Cell 之间的物理域隔离、供电切换，以及极限模式下的电源策略执行。

#### 1.5.2 双电池物理域隔离

Tree 设备内置两块物理独立的电池：

| 电池      | 容量                   | 供电对象                                                 | 可独立工作               |
| :-------- | :--------------------- | :------------------------------------------------------- | :----------------------- |
| 主电池    | 常规容量（依机型而定） | Trdom 主 CPU、Trneuro、Carbon M 显示芯片、屏幕、传感器等 | 否（依赖整体系统）       |
| PACT-Cell | 500 - 1000 mAh         | Trcomm S1、Trsec、Trpower 自身逻辑                       | **是（完全独立供电域）** |

**隔离机制**：

- 主电池与 PACT-Cell 在物理层通过 Trpower 内部的 MOSFET 隔离，输出端不共地。
- 主电耗尽后，Trpower 通过 PMIC 硬线信号（EMG_EN）切断主供电岛 A 岛，同时将 B 岛（PACT-Cell）专线输送给 Trcomm S1 和 Trsec。
- A 岛（主电池供能）：仅负责屏幕背光驱动与 AP 极简内核（时间显示 + 触摸中断）。
- B 岛（PACT-Cell 供能）：专线直供 Trcomm S1（射频收发）、Trsec（加密/签名）和麦克风偏置电路。

#### 1.5.3 3% 电量硬切换

当 Leaf OS 监测到主电池电压跌至 3% 阈值时：

1. Trpower 不经过 AP 软件调度，直接由 Trsec 拉高 PMIC 的 EMG_EN 引脚。
2. 内部高侧 MOSFET 将主板物理分割为 A/B 两个独立供电域。
3. Leaf OS 强制限制屏幕峰值亮度不超过 5 尼特，关闭所有 DC-DC 突发模式。
4. 射频发包瞬间（电流尖峰），Trpower 暂时切断屏幕供电毫秒级抖动，将主电池全部瞬时功率让给 Trcomm S1 的 PA（功率放大器）。

#### 1.5.4 极限生存模式（僵尸广播）

主电彻底耗尽、屏幕完全黑掉后：

- PACT-Cell 仍能让 Trcomm S1 以极低占空比（每 5 秒广播一次 200ms 信标）持续工作。
- Trsec 强制麦克风持续拾音，将压缩后的加密音频流缓存在 Trsec 独立 SPI Flash 中。
- 总续航时间：**6 - 8 小时**（视信标占空比与信道质量而定）。

此时 Tree 设备已物理黑屏，但内部仍是一颗“有声黑匣子”——持续广播最后位置，等待附近通信端路过。

### 1.6 显示芯片：Carbon M 系列

#### 1.6.1 职能定位

Carbon M 系列显示芯片负责 Tree 设备的屏幕渲染与输出控制。在正常模式下，它配合 Trdom 主 GPU 完成高帧率画面输出；在极限 SOS 模式下，它独立承担 **极简信息渲染** 任务，降低功耗。

#### 1.6.2 极限模式渲染策略

当进入 PACT 极限 SOS 模式（主电 3% 阈值触发）：

- 屏幕强制切换为**单色（黑白反色）**。
- 刷新率降至 **1Hz**（每秒刷新一次）。
- 显示内容仅包含硬编码的两个图层：
  - “紧急呼叫中”
  - “剩余待机时间”
- GPU 完全关闭，由 Carbon M 芯片内的 2D 硬加速器直接绘制。
- 触控层被切断，交互切换至物理按键（音量键被重映射为求救/挂断硬线中断）。

#### 1.6.3 与 Trneuro 的协同

Carbon M 与 Trneuro 之间有一条专用低带宽通道。当 Trneuro 完成 AI 音质修复后，若 MOS 分低于 3.0，Carbon M 可在屏幕上显示 **“音质已修复，可能非原始音色”** 的硬编码提示（不依赖 Leaf OS GUI 框架），用于现场救援人员快速判断音频可信度。

### 1.7 附加硬件：隐藏式增益天线与防拆走线

#### 1.7.1 可重构增益天线

天线本体隐藏于 Tree Pad/Phone 的金属镁铝合金中框内，作为辐射体的一部分。

- **孔径调谐器**：由 Trcomm S1 直接控制，在 PACT 模式下切换到高 Q 窄带谐振。
- **SAR（比吸收率）合规**：馈点处集成 Trsec 控制的接近传感器硬逻辑。检测到人体紧密接触时，硬件级回退发射功率（Back-off），但保持定向辐射能力（向设备背面发射，避开手持方向）。

#### 1.7.2 物理防拆蛇形走线

在主板上，包裹 Trsec 与 PMIC、Flash 之间关键信号线（EN 引脚、SPI 总线等）的蛇形走线构成物理防拆检测回路：

- 一旦主板被钻孔、撕开分层或飞线导致走线断裂，Trsec 检测到阻抗突变。
- 触发条件：断裂后，Trsec 立即向 FUSE_TAMPER 熔断位写入 1，**零化所有 PACT 根密钥**，设备永久无法恢复。
- 此举将物理攻击的代价抬升至“设备完全报废”级别。

**第一部分结语**：
硬件架构的每一颗芯片、每一条走线、每一个熔断位，都在回应同一个根本问题：**当一切不可控时，什么依然可信？** 我们的答案是——物理定律不可篡改，加密哈希不可逆，eFuse 熔断不可回滚。Tree 设备的硬件平台，正是在这三条铁律之上，搭建了从日常计算到临终求救的全部可能性。

下一部分将进入 **软件与系统架构**，详细描述 Leaf OS 的启动链、微内核设计、Hypervisor 隔离与分区布局。
